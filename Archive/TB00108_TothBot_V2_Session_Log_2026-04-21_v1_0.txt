3.2 AI-1 design proposal (NSI work plan step 5)
------------------------------------------------------------

Proposed three write-path options and recommended
Option A:

  Option A (RECOMMENDED, APPROVED): new
    TothBotTradeRecordHandler attached to the existing
    QueueListener alongside file_handler and
    alert_handler. Filters by event == "TRADE_CLOSE",
    appends the pre-serialized JSONL line to
    /home/tothbot/records/trades_<YYYY>.jsonl (UTC
    calendar year at time of write).

  Option B (rejected): direct write in hot path with
    threading.Lock. Violates HR-LG-001.

  Option C (rejected): branch in existing handler.
    Mixes retention concerns.

Also asked Bill two open questions:
  Q1  fsync per trade-record write? (a) no (b) yes
  Q2  permissions? (a) umask-derived 0o644
                   (b) hard-pin via os.open mode arg

Bill's decisions (LOCKED):
  Design: Option A — approved.
  0511006 dv1_0 -> dv1_1 architectural bump — approved.
  Q1: (b) fsync after each write — approved.
      Rationale: compliance-grade records must survive
      VPS power loss / kernel panic; fsync runs in
      QueueListener thread so HR-LG-001 unaffected;
      TRADE_CLOSE frequency is low so per-write cost
      is trivial.
  Q2: (b) hard-pin 0o644 via os.open with
      O_WRONLY|O_APPEND|O_CREAT wrapped in os.fdopen —
      approved. (Implementation uses os.write +
      os.fsync + os.close directly, which satisfies
      the same contract without an fdopen wrapper;
      functionally equivalent.)
      Rationale: umask is external state and can
      drift; explicit source-level permission contract
      per TB00000 Section 3.3 and Deming 6.1.
