4.1 DEFECT-001 — execution_engine.py line 255
------------------------------------------------------------

Old:  self._pm.create_record(symbol, cl_ord_id,
                              entry_limit_price, entry_qty)
Problem: PositionMirror has no create_record() method.
         WS Manager is SOLE WRITER per HR-PM-009.
         EE is calling wrong object, wrong method, wrong
         architecture.

Root cause: EE was written with a standalone pm object
            injected. At runtime, WM owns position_mirror
            as an internal dict of WM's own PositionRecord
            class. wm.add_order() BOTH sends the order AND
            creates the PM record.

Fix: Replace the entire entry message construction + send
     block (lines 233–265 in on_gate8_output) with a
     single call:
       await self._wm.add_order(
           symbol, entry_limit_price, entry_qty, cl_ord_id
       )
     Remove: manual add_order message dict construction
     Remove: await self._wm.ws_private.send(...)
     Remove: self._pm.create_record(...)
     Remove: self._wm.pending_orders[cl_ord_id] = ...
             (wm.add_order() handles pending_orders)
     Remove: self._wm.req_id_registry[req_id] = ...
             (not needed for entry — only batch_add)
     Keep:   cl_ord_id = _make_entry_cl_ord_id(symbol)
     Keep:   atr_14 computation
     Keep:   self._entry_orders[cl_ord_id] = {...} storage
     Keep:   minimum size validation (lines 212–226)
             (redundant but harmless — defense in depth)
     Keep:   ENTRY_DISPATCHED log

Note: wm.add_order() uses ENTRY_GTD_SEC=30s. EE used
      ENTRY_TIMEOUT_SEC=45s. The wm.add_order() value
      governs after this fix. CIATS owns this parameter.
      Starting value alignment is a CIATS concern.

4.5 DEFECT-005 — ws_manager.py batch_add ACK not handled
------------------------------------------------------------

Problem: Kraken batch_add response has method="batch_add".
         _recv_loop_private extracts channel via:
           msg.get("channel") or msg.get("method", "")
         This yields "batch_add" as the channel key.
         _channel_dispatch has no "batch_add" entry.
         Message goes to UNKNOWN_MESSAGE_TYPE and is
         discarded. EE.on_batch_add_response() is never
         called. tp_order_id and emergsl_order_id remain
         empty string on every fill.

Fix:
  (a) Add to _setup_dispatch_tables():
        "batch_add": self._handle_batch_add_ack,

  (b) Add new method _handle_batch_add_ack(msg):
        req_id = msg.get("req_id")
        if self._exec_engine_fn:
            await self._exec_engine_fn.on_batch_add_response(msg)
            # OR route via existing exec_engine_fn callback pattern

  NOTE: The exec_engine_fn callback pattern is
  Callable[[dict, WSManager], Awaitable]. EE.on_batch_add_response()
  takes msg: dict only (no wm arg). The handler must call
  the EE method correctly.

  Correct approach: store EE reference directly OR add
  a separate batch_add_ack_fn callback. Next session
  must check how exec_engine_fn is wired in startup_sequence
  and match the call signature.
