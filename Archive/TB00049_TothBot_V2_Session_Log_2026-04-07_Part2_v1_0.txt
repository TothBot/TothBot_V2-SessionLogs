2.2 CRITICAL PRE-CODING FINDING -- Field Name Mismatch
------------------------------------------------------------

Two discrepancies found that MUST be resolved before
test_integration.py can be written correctly.

FINDING 1 -- PositionRecord field name conflict:

  1011006 coding spec (Section 4) defines:
    cl_ord_id_entry      str
    entry_limit_price    Decimal
    entry_fill_price     Decimal
    emgsl_order_id       str
    emgsl_cl_ord_id      str

  ws_manager.py (line 136) PositionRecord dataclass has:
    cl_ord_id            str   (NOT cl_ord_id_entry)
    -- no entry_limit_price field present --
    entry_fill_price     Decimal
    emergsl_order_id     str   (NOT emgsl_order_id)
    emergsl_cl_ord_id    str   (NOT emgsl_cl_ord_id)

  Specifically:
    Spec uses:  cl_ord_id_entry   / ws_manager uses: cl_ord_id
    Spec uses:  emgsl_order_id    / ws_manager uses: emergsl_order_id
    Spec uses:  emgsl_cl_ord_id   / ws_manager uses: emergsl_cl_ord_id
    Spec has:   entry_limit_price / ws_manager: absent

FINDING 2 -- PositionMirror method resolution:

  ExecutionEngine._pm is injected as "position_mirror"
  and calls these methods:
    self._pm.create_record(symbol, cl_ord_id,
                           entry_limit_price, entry_qty)
    self._pm.on_entry_filled(symbol, entry_fill_price,
                             entry_qty, timestamp)
    self._pm.on_batch_add_ack(symbol, tp_order_id,
                              emgsl_order_id)
    self._pm.clear_record(symbol)

  These method names do NOT match the 1011006 spec
  method names:
    Spec uses:    create_position_record()
    EE uses:      create_record()

    Spec uses:    on_entry_filled()  (same name -- OK)
    Spec uses:    on_batch_add_ack() (same name -- OK)
    Spec uses:    close_position()
    EE uses:      clear_record()

  None of these methods exist in any uploaded .py file.
  PositionMirror is NOT a separate class in the repo.
  Position Mirror state is a plain dict on WSManager:
    self.position_mirror: dict[str, PositionRecord] = {}
  The methods in 1011006 spec are pseudocode-style
  operations, not a class API.

  This means: ExecutionEngine._pm must be a duck-type
  object (or WSManager itself) providing these methods.
  There is no position_mirror.py module in the repo.

------------------------------------------------------------
2.3 Impact Assessment
------------------------------------------------------------

These discrepancies block test_integration.py until resolved:

  (a) PositionRecord field names in ws_manager.py must match
      the names used by ExitController and CIATS at runtime.
      If exit_controller.py uses pos.emgsl_order_id but
      ws_manager.py stores emergsl_order_id, the code
      crashes at runtime — not at test time.

  (b) ExecutionEngine._pm.create_record() / .clear_record()
      are called at runtime. If no object provides these
      methods, execution_engine.py crashes on first entry
      dispatch.

  Resolution required (Bill to confirm):
    Option A: Add a PositionMirror wrapper class that
              provides create_record(), on_entry_filled(),
              on_batch_add_ack(), clear_record() as thin
              wrappers over the position_mirror dict on
              WSManager. ExecutionEngine._pm = this wrapper.

    Option B: Add these methods directly to WSManager and
              pass `wm` as the `position_mirror` arg to EE.

    Option C: The PositionRecord in ws_manager.py is wrong.
              Update it to match 1011006 spec field names.

  Claude cannot self-approve a resolution. Bill decides.
