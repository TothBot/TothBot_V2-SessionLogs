2.1 Bug Fix -- ws_manager.py _handle_canceled CASE C
------------------------------------------------------------

PROBLEM: _handle_canceled CASE C (cum_qty==0, no fill)
logged EXEC_CANCELED and popped pending_orders but did NOT
call exec_engine_fn. Position Mirror record created at
dispatch by wm.add_order() was never cleared on rejection.
BoundedSemaphore was never released.

FP/DP ANALYSIS:
  Option A (fix in WM directly): Eliminated. WM has no
  reference to RiskEngine — cannot release semaphore.
  Would duplicate cleanup logic already in EE. Two code
  paths for same concern. High future rework risk.

  Option B (call exec_engine_fn from CASE C): Selected.
  Consistent with established WM→EE contract pattern
  (_handle_filled already does this). EE._on_entry_canceled
  handles complete cleanup: pop pending_orders, clear PM,
  release semaphore. Non-entry cancels are no-ops in EE
  (cl_ord_id not in _entry_orders). Zero duplication.
  Zero future rework risk.

FIX APPLIED to ws_manager.py _handle_canceled CASE C:
  Added exec_engine_fn call passing canceled event dict.
  Same pattern as _handle_filled.
