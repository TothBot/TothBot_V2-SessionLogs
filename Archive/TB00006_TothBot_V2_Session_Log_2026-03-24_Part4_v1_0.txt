2.2 Nine Internal Architecture Improvements Applied
------------------------------------------------------------

Before the FLAG analysis, nine internal improvements
were identified through architecture review and applied
to 0500000. All improve the primary mission of maximizing
profit and minimizing loss with emphasis on minimizing
loss. None change trading logic — all improve reliability,
correctness, or computational efficiency.

  I-1 — Gate 3 Regime Pre-Filter (NEW gate)
    Problem: Pipeline evaluated SSS signal engine even
    for TRENDING_NEG and NON_DIR+ELEVATED regimes, which
    are definitive NO ENTRY conditions. Wasted computation
    on every candle for every pair in those regimes.
    Solution: Add Gate 3 Regime Pre-Filter immediately
    after Gate 2 Liquidity Gate. Reads from regime cache
    (already maintained). Blocks TRENDING_NEG (any
    volatility) and NON_DIR+ELEVATED before reaching SSS.
    Only NON_DIR+NORMAL passes Gate 3 to SSS (sized 50%
    by Gate 6). Gate numbering advanced from 7 to 8 gates.
    Old Gate 3 (1H HTF) becomes Gate 4.
    Old Gate 4 (Selection Controller) becomes Gate 5.
    Old Gate 5 (Regime Gate) becomes Gate 6 — now handles
    only NON_DIR+NORMAL 50% sizing.
    Old Gates 6 and 7 become Gates 7 and 8.
    Impact: Eliminates all SSS computation on no-trade
    regimes. No change to trading decisions.

  I-2 — Pre-Computation Cache Architecture
    Problem: Pipeline architecture did not explicitly
    document that indicator values are pre-computed and
    cached — not computed on demand during pipeline
    evaluation. The hot path must never compute.
    Solution: Add Section 2.5 to 0500000 documenting the
    pre-computation cache. Four cached values: regime
    classification (from REST GetOHLCData daily — FLAG 2),
    1H EMA(20)/EMA(50) state (from ohlc(60) — FLAG 1),
    ATR(14) per pair (incremental — I-9), and 24h
    liquidity (daily). Hot path reads cache only.
    All computation occurs in update handlers outside
    the hot path.

  I-3 — Position Mirror O(1) Lookup
    Problem: Position Mirror structure not specified. If
    implemented as a list, lookup requires O(N) scan on
    every ticker event. With 20 concurrent positions and
    continuous ticker events, this degrades under load.
    Solution: Position Mirror is a symbol-keyed dict.
    O(1) lookup by symbol on every ticker event. AR-002
    updated with this specification.

  I-4 — Parameter Store Snapshot Pattern
    Problem: CIATS owns all parameters and may update
    them. If CIATS applies a parameter change mid-pipeline
    (e.g., per_trade_pct changes between Gate 7 and Gate 8
    during the same evaluation), the pipeline has an
    inconsistent parameter set. This is a correctness
    defect that produces non-deterministic behavior.
    Solution: TothBot takes a frozen snapshot of all
    CIATS parameters at the start of each pipeline
    evaluation. The pipeline uses the snapshot throughout.
    CIATS writes are queued and applied only at confirmed
    inter-trade boundaries. AR-014 added. Assign to
    0511002 (pipeline) and 0211001 (CIATS).

  I-5 — Logger Async Bounded Queue
    Problem: Logger architecture did not specify write
    path behavior. If Logger.write() blocks the calling
    thread on a slow disk write, it directly blocks the
    hot path. One slow disk write stops the entire
    pipeline. This is a loss-causing defect.
    Solution: Logger writes go to an in-memory bounded
    queue. A separate async writer thread flushes to disk.
    Logger.write() returns immediately — never blocks
    the hot path. Queue must be bounded. Immediate alert
    if queue fills (indicates disk write is falling behind).
    AR-014 added. Assign to 0511006.

  I-6 — Layer 2 Cancel Timeout Fallback
    Problem: Layer 2 MAE exit sends cancel_order for TP
    and emergSL before issuing the market sell. If a cancel
    ACK is never received (WS lag, drop, etc.), the
    current architecture had no specified behavior. Issuing
    a market sell with an unacknowledged cancel means a TP
    or emergSL order may still be active when the market
    sell executes — creating a position-doubling risk or
    incorrect P/L calculation.
    Solution: req_id registry (already specified in A-2)
    tracks each cancel ACK. On cancel timeout:
      Step 1: Check executions channel state for the order.
      Step 2a: Order confirmed cancelled — proceed.
      Step 2b: Order state unknown — retry cancel once.
      Step 3: Second timeout — ALERT, hold position, alert
               operator immediately. Do NOT issue market
               sell with ambiguous order state.
    NEVER market sell with ambiguous order state — this is
    an absolute requirement. Added to Section 4.1 Layer 2
    table, Section 11.2. Assign to 0511001 (WS Manager
    timeout behavior) and 0511004 (Exit Controller logic).
    OI-003 updated with this requirement.

  I-7 — Ticker-Driven Continuous Drawdown Monitoring
    Problem: Gate 7 Risk Guard drawdown check only ran
    at each 5-minute candle close. An adverse price move
    between candle closes was not detected by Risk Guard
    until the next pipeline evaluation — up to 5 minutes
    later. Ticker events are already subscribed and
    arrive continuously. Not using them for drawdown
    monitoring leaves a 5-minute detection gap.
    Solution: Gate 7 Risk Guard is extended to also
    evaluate drawdown on every ticker event for held
    positions. No new subscriptions required — ticker
    already present. The continuous check uses the same
    drawdown thresholds as the candle-close check. This
    reduces maximum drawdown detection latency from
    5 minutes to sub-second.

  I-8 — WS Message Dispatch Table
    Problem: WS Manager message routing not specified
    architecturally. If implemented as an if/elif chain
    on channel and message type, every message traverses
    the full chain to find its handler. O(N) routing
    where N = number of message types. This is
    inefficient and adds latency to every message.
    Solution: WS Manager routes all incoming messages via
    a dispatch table keyed on (channel, message_type).
    O(1) routing regardless of message count. AR-015
    added. Assign to 0511001.

  I-9 — ATR(14) Incremental Running Calculation
    Problem: ATR(14) computation not specified. If
    recomputed from scratch on each pipeline evaluation,
    it requires accessing the last 14 candles, computing
    True Range for each, and averaging. O(14) computation
    per pipeline eval per pair. With 20 pairs this is
    O(280) per eval. Unnecessary — ATR can be maintained
    as an incremental running value.
    Solution: ATR(14) is maintained as an incremental
    running calculation. On each 5m candle close event,
    the new ATR value is computed from the prior ATR value
    and the new candle's True Range (standard Wilder
    smoothing). O(1) per update. Hot path reads the
    cached value. AR-016 added. Assign to 0511002 and
    0511004.

2.4 FP/DP Analysis — FLAG 1 and FLAG 2
------------------------------------------------------------

Three sequential searches executed. 8+ sources confirmed.
Minimum 5 sources per finding per TB00000 Section 5.3.

SEARCH 1: Kraken WebSocket v2 OHLC channel intervals
  Sources: Kraken API Center OHLC docs, Kraken WS API
  FAQ, WS v2 change log, node-kraken-api TypeScript
  client, python-kraken-sdk docs, AsyncAPI spec example,
  Kraken WS v1 docs, AsyncAPI blog post.
  Confirmed: WS v2 ohlc channel supports intervals
  1|5|15|30|60|240|1440|10080|21600 minutes. All on
  same Public WS v2 connection. No new connections
  required for ohlc(60) alongside ohlc(5m). Both fire
  on trade events, not wall clock — same stall behavior.

SEARCH 2: WS ohlc vs REST for different use cases
  Sources: Kraken API Center GetOHLCData docs, Kraken
  WS FAQ, node-kraken-api, CoinAPI blog on real-time
  data challenges, REST vs WS architectural patterns.
  Confirmed: REST GetOHLCData returns up to 720
  committed candles at the time of the call. The last
  entry in the REST response is always the current
  not-yet-committed candle. WS ohlc at long intervals
  has the same stall-until-next-trade behavior. At a
  daily interval, the WS snapshot on reconnect may
  deliver an old (not yet closed) candle — creating
  ambiguity about whether the data is the just-closed
  daily candle or the currently opening one.

SEARCH 3: Multi-timeframe architecture patterns for
  event-driven trading systems
  Sources: NautilusTrader architecture docs, CoinAPI
  blog on trading engine data challenges, FMZQuant
  multi-timeframe EMA strategy, multi-timeframe mean
  reversion system, FMP real-time data API guide.
  Confirmed: Industry pattern for higher-timeframe
  indicators in event-driven systems is to cache them
  and update from the appropriate event source.
  Scheduled batch reads (REST) are appropriate for
  computations that run on fixed schedules.
  Event-driven updates (WS) are appropriate for
  real-time responsive caches.

FINDINGS FROM FP/DP ANALYSIS:

  FLAG 1 — RECOMMENDATION: WS ohlc(interval=60) on
  the existing Public WS v2 connection.

  Reasoning:
  (1) Stays fully event-driven. No timers. No polling.
      No new connections. Subscribe an additional
      interval on the existing channel.
  (2) The $500k/day liquidity gate already eliminates
      illiquid pairs where the stall risk would be
      material. ohlc(60) stall on an illiquid pair means
      no trades — Gate 2 would have blocked the pair.
  (3) EMA(20) and EMA(50) on 1H candles are among the
      slowest-moving indicators in the system. A 1H
      candle arriving 2 minutes late produces an EMA
      value mathematically indistinguishable from one
      computed at exactly the 1H mark.
  (4) REST alternative requires a 1H timer/scheduler —
      direct violation of the event-driven principle.
  (5) Internal accumulation from 5m candles is more
      complex code with more failure modes for no benefit.

  FLAG 2 — RECOMMENDATION: REST GetOHLCData called
  once per monitored pair at 00:00 UTC.

  Reasoning:
  (1) Regime computation is a scheduled batch operation,
      not a response to a market event. The event-driven
      principle governs real-time trading operations.
      Calling REST once per pair per day at a scheduled
      time is architecturally correct — this is what
      REST is for.
  (2) WS ohlc(1440) introduces open/committed candle
      ambiguity at the interval boundary. At 00:00 UTC,
      TothBot needs the just-closed daily candle. REST
      GetOHLCData at 00:00 UTC returns 720 clean
      committed candles unambiguously. No boundary logic
      required.
  (3) Rate counter impact is negligible: 20 pairs x 1
      call/day = 0.014 calls/minute vs threshold of
      125/pair. Invisible to rate counter.
  (4) ADX(14) and EMA(20/50) on daily candles are
      among the slowest-moving indicators in the system.
      A 2-minute timing difference at midnight cannot
      change the regime result. The open/committed
      ambiguity question is moot for precision — it is
      an architectural correctness question.

  This is also the fourth legitimate use of REST:
    (1) GetWebSocketsToken — startup/reconnect
    (2) GetOpenOrders + GetAccountBalance — restart fallback
    (3) CancelAllOrders — kill-switch emergency
    (4) GetOHLCData — daily regime batch computation NEW
