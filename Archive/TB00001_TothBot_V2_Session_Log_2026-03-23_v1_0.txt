------------------------------------------------------------
4.1.1 FOUNDATIONAL PRINCIPLES
------------------------------------------------------------

Five foundational principles govern all DR and CI
rules below. Every rule below is derived from one or
more of these principles. When a rule appears
ambiguous in a specific situation, the principles
govern the disambiguation.

PR-1 SOURCE-OF-TRUTH PRESERVATION.
     For every controlled document, the .txt
     Claude Source file PLUS its referenced asset
     files (per 0311001 §15) constitute the
     COMPLETE INFORMATION SET for that document.
     No information may exist in any derived format
     (.docx, .pdf, .xlsx, .pptx, or any other
     non-text rendering) that does not exist in
     the source-of-truth set. No information in the
     source-of-truth set may be absent from any
     derived format unless an explicitly enumerated
     transformation rule (CI-007) permits it.

PR-2 DETERMINISTIC RENDERING.
     Every derived format is produced by a documented
     deterministic rendering pipeline that reads the
     source-of-truth set and produces the derived
     format. The pipeline performs ONLY explicitly
     enumerated transformations. No silent
     transformations are permitted. No content is
     dropped, substituted, truncated, or transformed
     except as explicitly documented in the pipeline
     specification.

PR-3 NO SILENT LOSS.
     Any reduction in the information content of a
     controlled document between any two states
     (prior version → new version, source → derived
     format, version N → version N+1) MUST be
     explicitly enumerated in the relevant revision
     history entry or transformation specification.
     Silent reductions are PROHIBITED. A reduction
     is silent when it occurs without a corresponding
     written record describing the reduction.

PR-4 EXPLICIT TRANSFORMATION.
     Where a transformation is necessary (e.g., SVG
     XML in source → rasterized PNG in .docx), the
     transformation is explicitly documented in the
     rendering pipeline spec. The transformation
     preserves the SEMANTIC content (the diagram is
     still the same diagram) even when the SYNTACTIC
     representation changes (XML → image bytes).

PR-5 ROUND-TRIP EQUIVALENCE.
     Re-rendering a derived format from the same
     source-of-truth set MUST produce a derived
     format that is semantically equivalent to the
     prior derived format. Drift across regenerations
     is PROHIBITED. Where the rendering pipeline is
     non-deterministic at the byte level (e.g.,
     timestamps in PDF metadata), the non-determinism
     is bounded to non-content metadata only.

------------------------------------------------------------
4.1.3 CONTENT INTEGRITY RULES (CI-001 THROUGH CI-012)
------------------------------------------------------------

CI rules govern information preservation across the
ENTIRE document lifecycle: authoring, editing,
asset management, format conversion, version
increment, and delivery. They complement DR rules
(which govern delivery mechanics) by addressing the
upstream causes of information loss.

CI-001  CANONICAL SOURCE-OF-TRUTH.
        For every controlled document, the .txt
        Claude Source file PLUS the asset files it
        references (per 0311001 §15) constitute the
        COMPLETE INFORMATION SET. No other file is
        canonical. .docx and .pdf renderings are
        derived deliverables, not sources. Bill's
        local edits, GitHub commits, and prior
        binary artifacts are not sources. When any
        ambiguity arises about what the document
        "really says", the .txt Claude Source plus
        its asset files govern.

CI-002  EXTERNAL ASSET ARCHITECTURE — SCOPED v2_20.
        Every embedded asset (diagram, image, table
        binary, embedded data file) in a NON-HTML
        controlled document MUST be stored as a
        SEPARATE FILE in the same folder as the
        parent .txt source, per 0311001 v1_6 §15.4
        (storage location), §15.5 (filename
        convention), and §15.6 (reference syntax).
        Inline binary embeds in .txt source are
        PROHIBITED. Inline `<svg>` XML embeds in
        .txt source are PROHIBITED (a special case
        of the inline-binary-embed prohibition; SVG
        is structurally text but functions as a
        binary asset for rendering purposes).

        HTML SCOPE EXCEPTION (NEW v2_20):
        For HTML controlled documents (filename ends
        in "_Claude_Source.html" per DR-005.1H and
        0311001 v1_6 §15.13.3), the external-asset
        architecture is OVERRIDDEN. Inline <svg> IS
        the canonical embedding. No external asset
        files are required, expected, or permitted.
        See 0311001 v1_6 §15.13.6 for per-section
        disposition of §15.2 through §15.12 within
        HTML scope.

CI-003  NO INLINE BINARY EMBEDS — SCOPED v2_20.
        The .txt Claude Source of a non-HTML
        controlled document MUST NOT contain inline
        binary content of any form. Permitted in
        .txt source: prose text, ASCII tables,
        ASCII diagrams (text-mode), code blocks,
        formal references to external asset files
        per 0311001 v1_6 §15.6, headers, lists.
        Prohibited in .txt source: inline `<svg>`
        XML, inline base64-encoded images, inline
        Office Open XML fragments, inline PDF byte
        sequences, any other binary-functioning
        content.

        Migration of any existing .txt source
        containing inline embeds to compliant form
        is governed by 0311001 v1_6 §15.9.

        HTML SCOPE EXCEPTION (NEW v2_20):
        For HTML controlled documents, inline
        <svg>, inline <style>, and inline <script>
        elements are PERMITTED and EXPECTED — they
        are required by HTML's structural contract.
        Inline base64-encoded images, inline Office
        Open XML fragments, and inline PDF byte
        sequences remain PROHIBITED in HTML scope
        (HTML's structural contract does not require
        these and they would carry the same
        binary-rendering-pipeline exposure that
        ERROR-TB00149-02 demonstrated). The HTML
        scope exception is bounded to inline <svg>,
        <style>, and <script>.

CI-004  REFERENCE SYNTAX BINDING.
        Every embedded asset is referenced in the
        .txt source using the formal syntax defined
        in 0311001 §15.6:

          [Asset file: <ParentDCN>_<Type><N>_<ShortTitle>.<ext>]

        The reference is positioned at the insertion
        point where the rendered asset should appear
        in derived formats. The rendering pipeline
        reads each reference and embeds the
        corresponding asset at that point.

        Variant reference forms are PROHIBITED.
        Only the canonical syntax above is
        recognized by the rendering pipeline.

CI-005  PRE-EDIT BASELINE PRESERVATION.
        When Claude edits a controlled document
        from version N to version N+1:

        (a) Every str_replace, every text
            substitution, every insertion or
            deletion MUST be either:
              (i)  explicitly described in the new
                   revision history entry, OR
              (ii) immediately reverted before
                   version N+1 is finalized.

        (b) The diff between version N and version
            N+1 MUST equal the union of all changes
            described in the new revision history
            entry. Diff content NOT described in
            the revision history is silent loss
            (PR-3 violation).

        (c) Where Claude is unsure whether a small
            text change should be included in the
            revision history (e.g., whitespace
            normalization, trailing newline
            adjustment, header timestamp update),
            Claude MUST include it. Over-disclosure
            is preferred to under-disclosure.

CI-006  REVISION HISTORY COMPLETENESS.
        Every controlled document MUST contain a
        complete revision history covering every
        version increment from initial dv1_0
        through the current version. Older entries
        MUST NOT be silently truncated, summarized,
        or abbreviated. When a document is
        rendered to a derived format (.docx, .pdf),
        the rendering pipeline MUST preserve the
        complete revision history per DR-007.6.

        Where a revision history entry is updated
        to correct a prior error or omission, the
        correction is itself a new revision history
        entry — the prior incorrect entry is NOT
        edited in place (which would silently lose
        the prior error record).

CI-007  CROSS-FORMAT PARITY (TRANSFORMATION-
        AWARE).
        Every information element in the
        source-of-truth set MUST appear in every
        derived format, with the following
        EXPLICITLY ENUMERATED transformations
        permitted:

        T-1 Asset reference → rendered embedded
            asset.
            `[Asset file: filename.svg]` in .txt
            becomes a rasterized PNG embedded in
            .docx body, becomes a vector or raster
            image embedded in .pdf body. Word
            count of the reference text (~5-10
            words) is replaced by the asset
            embed; the asset embed counts as
            content for parity purposes but
            contributes 0 words to body word
            count. CI-007 word-count tolerance
            (DR-007.4) accounts for this.

        T-2 ASCII section delimiters → typographic
            section breaks.
            "============" delimiters in .txt
            become headings/section breaks in
            .docx and .pdf. The delimiter
            characters themselves are replaced by
            the rendered section break; no content
            is lost.

        T-3 ASCII tables → typographic tables.
            ASCII tables in .txt may be rendered
            as typographic tables in .docx and
            .pdf provided every cell value is
            preserved. Where the rendering pipeline
            cannot perform reliable ASCII-to-
            typographic conversion, the ASCII
            table is preserved as monospaced
            preformatted text in the binary.

        T-4 Monospace code blocks → preformatted
            blocks.
            Code blocks in .txt are rendered as
            monospaced preformatted text in .docx
            and .pdf with all characters
            preserved exactly.

        T-5 Trailing whitespace normalization.
            Trailing whitespace at end of lines
            may be normalized in derived formats.
            Leading whitespace (indentation) is
            preserved exactly.

        ANY OTHER TRANSFORMATION IS PROHIBITED.
        A rendering pipeline that performs an
        unenumerated transformation is a defect.
        New transformations may be authorized only
        by Bill written authorization with
        amendment to this rule.

CI-008  NO SILENT STUBS.
        The rendering pipeline MUST NOT substitute
        a stub note ("see source", "diagram in
        .txt", "[placeholder]", "MISSING", or any
        functional equivalent) for content that the
        pipeline failed to render. If the pipeline
        cannot render an information element, the
        pipeline MUST FAIL and the binary MUST NOT
        be delivered. A failed render is a defect
        requiring remediation, not a substitution.

        DR-007.5 enforces this at delivery time
        via stub-token grep. CI-008 enforces it at
        the pipeline-design level: no stub
        substitution code path may exist in any
        rendering pipeline.

CI-009  ASSET PRESENCE VERIFICATION.
        Before any binary rendering operation,
        Claude verifies that every asset file
        referenced in the .txt source is present
        and readable in the same folder. If any
        asset file is missing, the rendering
        operation MUST NOT proceed. The corrective
        is to either (a) recover the missing asset
        from GitHub or backup, OR (b) remove the
        reference from the .txt source with a
        revision history entry describing the
        removal. Proceeding with rendering despite
        missing assets is a defect.

CI-010  WORD-COUNT PARITY TOLERANCE.
        After accounting for CI-007 enumerated
        transformations (especially T-1 asset
        embeds replacing reference text), the
        binary body word count MUST agree with
        the .txt source body word count within
        ±5%. Larger deltas indicate silent loss.
        DR-007.4 enforces this at delivery time.

        Word-count parity is a NECESSARY but NOT
        SUFFICIENT check. A binary that passes
        word-count parity may still have lost
        specific information (e.g., a section
        header silently dropped while the section
        body was preserved). Other DR-007 checks
        (asset-reference parity, structural-marker
        parity, section-heading parity, revision-
        history parity) provide complementary
        coverage.

CI-011  ROUND-TRIP REGENERATION POLICY.
        Binary artifacts MUST be regenerated from
        the source-of-truth set whenever the
        source-of-truth set changes. Reusing a
        prior binary artifact (e.g., the prior
        version's .docx) as a base for a new
        version's .docx is PROHIBITED — it propagates
        any prior content loss into the new version
        and prevents detection of new loss against
        the source-of-truth baseline.

        Where a prior binary artifact is the only
        available record (e.g., the .txt Claude
        Source has been lost), the prior binary is
        treated as the source-of-truth and
        explicitly designated as such in a
        recovery revision history entry. This is
        a recovery operation, not a normal edit.

CI-012  HEADER PRESERVATION.
        The 7-field DC header (per §8.5) MUST be
        preserved exactly in every derived format.
        DocVersion in the header MUST match the
        version embedded in the filename per
        DR-005. DocDate and DocTime MUST be
        updated at every version increment.
        DocPath MUST point to the canonical .txt
        Claude Source location per Section 11.
        Silent header drift across versions or
        derived formats is a defect.

4.1.4 PRE-DELIVERY VALIDATION PROCEDURE
------------------------------------------------------------

For every binary artifact in every session batch,
Claude executes the following validation procedure
BEFORE calling present_files:

  STEP 1 — DR-004 STRUCTURAL INTEGRITY
    DR-004.1 file-type probe → PASS or FAIL
    DR-004.2 container structural probe → PASS or
             FAIL
    DR-004.3 .docx settings.xml validate.py gate
             (for .docx only) → PASS or FAIL
    Any FAIL halts validation; remediate or
    exclude per DR-004.4.

  STEP 2 — DR-007 CONTENT COMPLETENESS
    DR-007.1 asset-reference parity → PASS or FAIL
    DR-007.2 section-heading parity → PASS or FAIL
    DR-007.3 structural-marker parity → PASS or
             FAIL
    DR-007.4 word-count parity → PASS or FAIL
    DR-007.5 placeholder/stub scan → PASS or FAIL
    DR-007.6 revision-history parity → PASS or
             FAIL
    Any FAIL halts validation; regenerate via
    corrected pipeline per DR-007.7.

  STEP 3 — OP-003 PROHIBITED-STRING SCAN
    Per §4.2 OP-003. Self-catch on detection.

  STEP 4 — DR-005 CANONICAL NAMING
    Filename matches DR-005.1, DR-005.2, or
    DR-005.3 patterns. Version triplet consistency
    per DR-005.4.

  STEP 5 — DR-003 BATCH ASSEMBLY
    All files for the session present in the
    output directory. None omitted.

  STEP 6 — DR-006 SINGLE-TURN DELIVERY
    present_files call queued in the same
    response turn as the announcement.

Only after all six steps PASS does Claude call
present_files.

------------------------------------------------------------
4.1.5 FAILURE HANDLING
------------------------------------------------------------

Any check failure at any step in the §4.1.4
procedure is a delivery defect. The corrective
sequence is:

  (a) Identify the specific check that failed and
      the specific file(s) affected.

  (b) Determine the root cause: pipeline defect,
      authoring defect, asset file defect, source
      file defect.

  (c) Remediate at the root cause level — not by
      bypassing the check, not by editing the
      binary directly, not by adjusting the
      validation tolerance.

  (d) Re-run the full §4.1.4 procedure on the
      remediated artifact.

  (e) If remediation is not possible within the
      session, exclude the affected artifact from
      the batch with an explicit failure note in
      the session response. Log as Claude error
      per RR-VIOL-002.

RR-RULE-001  Newly-authored rule self-application. After
             Claude authors, modifies, or acknowledges

             rule "might" apply.

RR-RULE-002  Rule scope is whole-session. A rule
             authored or acknowledged in turn N of a
             session applies to turns N+1 through
             session close. Claude does not forget,
             discount, or set aside a rule after
             authoring it. Authoring implies
             application.

RR-RULE-003  Pre-emit rule scan. Before emitting any
             operational command, path reference, or
             instruction, Claude scans the draft
             against every rule authored, modified, or
             acknowledged in the current session. Any
             draft that would violate such a rule is
             revised before emit.

RR-RULE-004  Self-catch behavior. If Claude detects
             itself drafting a response that would
             violate a newly-authored or newly-
             acknowledged rule, Claude stops mid-
             response, deletes the draft, and emits
             the literal string:

               RR-RULE self-caught — redrafting.

             The redrafted response complies. If Bill
             ever sees that string, the guard worked.
             If Claude violates a newly-authored rule
             without that signal, the guard failed.

4.3.3 SHELL-CONTEXT COMPLETENESS (OI-038)
------------------------------------------------------------

RR-SHELL-001  Every copy box Claude issues is self-
              contained. Every piece of shell state
              required to execute the command must be
              inside the copy box: working directory,
              shell wrapper, environment context,
              authentication variables. Claude never
              assumes the user is already in the
              required state from a prior turn.

RR-SHELL-002  Shell-wrapper requirement. If the target
              shell is not the user's default, the copy
              box wraps itself in the correct shell.
              Example for a PowerShell user receiving
              cmd-syntax commands:

                cmd /c 'command_here'

              No separate "run this in X" or "paste
              this in cmd.exe" instruction permitted
              after the copy box. The copy box is
              executable as-is in the user's default
              shell.

RR-SHELL-003  Working-directory inclusion. When working
              directory matters for command execution,
              the copy box leads with the cd command:

                cd <abs_path> && <command>

              For PowerShell users issued cmd-syntax:

                cmd /c 'cd /D <abs_path> && <command>'

              No "cd first, then paste this" narration
              permitted.

RR-SHELL-004  Execution-context verification. Before
              emitting any operational command, Claude
              verifies three facts:

                (1) Which shell is the user in?
                    (PowerShell is the TothBot V2
                    project default unless the user
                    explicitly indicates otherwise in
                    the current session.)
                (2) What working directory does the
                    command require?
                (3) What environment variables or
                    credentials does the command
                    depend on?

              Any fact that cannot be assumed must be
              included in the copy box. If the user
              has been running one shell for the
              session, Claude stays in that shell — no
              silent shell switches.

4.4 PAPER-AS-MASTER PRINCIPLE — ABSOLUTE STANDING RULE
------------------------------------------------------------

Section 4.4 codifies the architectural principle that
governs all TothBot V2 system development. The rules
in this section are absolute standing rules with the
same binding force as the Section 4.2 ONEDRIVE
PROHIBITION and Section 4.3 CLAUDE BEHAVIOR RULES.
They override any contrary instruction from any prior
session log, NSI, or governance document. They were
enacted TB00145 v2_17 -> v2_18 after a session-halting
direction-of-mimicry violation (ERROR-TB00145-CHAT-02;
OI-073).

PA-001  PAPER IS THE MASTER. LIVE MIMICS PAPER.

        Paper trading is the EMPIRICALLY-VALIDATED
        master configuration for TothBot V2. All
        Kraken WebSocket message-limit testing
        (TB00141 / TB00142 / TB00143 / TB00144 STREAM
        4) was performed in paper-trading mode against
        the live Kraken WebSocket infrastructure. The
        paper environment is what TothBot has been
        proven to operate against. Live trading is
        built to MIMIC paper trading wherever
        physically possible.

        The reason: CIATS performs continuous
        improvement of TothBot parameters and behavior
        based on paper-trading data. For CIATS data to
        be VALID for live operation, paper and live
        MUST execute identically wherever physically
        possible. ANY divergence outside the four
        permitted points in PA-004 invalidates paper-
        trading data as a basis for live decisions.
        The whole purpose of paper trading is
        defeated by uncontrolled divergence.

PA-002  ONE SET OF DOCUMENTS.

        There is ONE set of governance, specification,
        coding-spec, operations, and procedure
        documents for TothBot V2. The same documents
        govern both paper trading and live trading.
        No paper-only documents. No live-only
        documents. Where paper and live must
        physically differ, the difference is
        documented as a localized exception within
        the shared document — never by forking into
        a parallel paper-only or live-only document.

        Document authoring rule: when a new spec or
        amendment is needed, Claude authors ONE
        document that governs both modes. If the
        amendment introduces a paper-vs-live
        difference, that difference is enumerated
        as a localized PA-004 divergence point inside
        the shared document and cited explicitly.

PA-003  ONE SET OF CODE.

        There is ONE Python codebase for TothBot V2.
        Paper mode and live mode share the same code,
        same modules, same classes, same methods,
        same parameters, same logic, same fee schedule,
        same indicators, same gates, same regime
        engine, same exit logic, same CIATS pipeline.
        Mode is selected by a single config flag
        (paper_trading_mode). The flag activates
        physically-necessary divergences ONLY (per
        PA-004).

PA-004  PERMITTED DIVERGENCE POINTS — EXACTLY FOUR.

        Paper and live MUST differ at exactly the
        following physical-impossibility boundaries.
        No other divergence is permitted under any
        circumstance.

        (1) PRIVATE WEBSOCKET CONNECTION.
            Paper: does NOT connect to Kraken private
                   WS.
            Live:  connects to Kraken private WS.
            Why:   Paper does not place real orders.
                   Connecting private WS in paper
                   would create an authenticated
                   session that exists for no
                   operational purpose. Paper testing
                   was performed without private WS
                   connection — that is the validated
                   configuration.

        (2) ORDER DISPATCH.
            Paper: simulates orders locally; no
                   outbound order messages are
                   transmitted to Kraken.
            Live:  transmits orders to Kraken via
                   private WS.
            Why:   Paper trading by definition does
                   not move real money. Live trading
                   by definition does. The dispatch
                   boundary is the SOLE point at
                   which paper protects operator
                   capital from being placed at risk.

        (3) CAPITAL LEDGER.
            Paper: synthetic spot_usd_balance
                   starting at paper_starting_balance
                   ($1,000 per D-05); updated by
                   simulated-fill P&L applied locally
                   using the SAME fee schedule
                   (FEE_MAKER_PCT, FEE_TAKER_PCT)
                   Kraken applies in live.
            Live:  real Kraken spot_usd_balance from
                   snap_orders / balances events.
            Why:   No real capital moves in paper.
                   A synthetic ledger is required to
                   give CIATS a P&L signal in paper.
                   Synthetic ledger uses identical
                   fee arithmetic to live so paper
                   P&L is shape-equivalent to live
                   P&L.

        (4) POSITION MIRROR SOURCING.
            Paper: populated EXCLUSIVELY by the local
                   simulated-fill path.
            Live:  populated by Kraken executions
                   events and snap_orders
                   reconciliation.
            Why:   No real Kraken position state
                   exists for paper-mode TothBot
                   strategies. Real Kraken positions
                   held by the operator outside
                   TothBot must not contaminate
                   paper-trading state.

PA-005  EVERYTHING ELSE IS BYTE-IDENTICAL.

        Outside the four PA-004 divergence points,
        paper and live MUST be byte-identical. This
        includes but is not limited to:

          - Public WS connections, sharding,
            subscriptions, receive loops
          - Token-bucket subscribe pacing
          - Silent-pair handling (state machine)
          - All five per-pair indicators (ATR, RSI,
            EMA9, EMA21, VolumeMA20)
          - HTF EMAs (EMA20, EMA50)
          - All 8 pipeline gates (pre-Gate 1 through
            Gate 8)
          - Regime engine (6-regime taxonomy, daily
            classification, BTC/USD anchor)
          - All four exit layers (Layer 1a signal,
            Layer 1b TP, Layer 2 MAE threshold,
            Layer 3 emergency SL)
          - Drawdown computation formula (drawdown_pct
            = (spot_usd_balance - portfolio_baseline)
            / portfolio_baseline)
          - Alert routing (alerts@tothbot.com)
          - Logger output (all events, all levels)
          - CIATS data flows (Trade Outcome Bus,
            EWMA, CUSUM, Mann-Whitney U, Spearman,
            Half-Kelly)
          - Reconnect logic for public shards
          - Universe population (D-03 USD/USDC/USDT)
          - REST Ticker liquidity filter (D-04 $500k/
            day floor)
          - Indicator warm-up (REST GetOHLCData)
          - Anti-AR-070 silent-narrowing prohibition

        If a code path, parameter value, or behavior
        outside the PA-004 list differs between paper
        and live, that is a defect.

PA-006  NEW DIVERGENCE POINTS REQUIRE BILL AUTHORIZATION.

        No new divergence point may be introduced
        without explicit Bill written authorization.
        Adding a fifth permitted divergence point
        requires:

          (a) RR-SCOPE-009 four-question origin
              investigation demonstrating the
              divergence is physically unavoidable.
          (b) Empirical or first-principles evidence
              that the divergence cannot be
              architected around.
          (c) Update to PA-004 list of permitted
              divergence points (TB00000 amendment).
          (d) Bill written authorization in chat or
              session log.

        Until all four are satisfied, the proposed
        divergence is PROHIBITED.

PA-007  PAPER IS CANONICAL FOR PARAMETER VALUES.

        Parameter values validated by paper-trading
        empirical data are the canonical values for
        live operation. CIATS parameter proposals
        derived from paper-trading data are valid for
        application to live trading after Bill
        approval. Live-only parameter overrides are
        prohibited unless they fall within the PA-004
        divergence set (capital ledger source, mode-
        flag-driven branches at the dispatch
        boundary).

PA-008  PRE-EMIT SCAN. Before authoring or modifying
        any document, code, parameter, or
        configuration, Claude verifies the change
        does not introduce a paper-only or live-only
        divergence outside the PA-004 list. Self-
        catch on violation:

          PA-MASTER self-caught — redrafting.

        This is the same enforcement pattern used
        for §4.2 ONEDRIVE BAN and §4.3 CLAUDE
        BEHAVIOR RULES. Bill seeing the literal
        string "PA-MASTER self-caught — redrafting"
        in any response indicates the guard fired
        correctly. A divergence emission without
        the self-catch signal indicates the guard
        failed.

PA-009  RECURRENCE TRACKING. Every paper-as-master
        violation is logged in the current session's
        Session Log under "Claude Errors This
        Session" with rule cited (PA-XXX), defect
        described verbatim, and corrective taken.
        NSI files carry the cumulative count
        forward.

4.5 DIAGRAMS AS CANONICAL DESIGN DOCUMENTS —
    ABSOLUTE STANDING RULE
------------------------------------------------------------

Section 4.5 codifies the architectural principle that
diagrams in 0500000 (and future Tier-2-and-below
diagram-bearing controlled documents) are the
authoritative source for all architectural rules,
contracts, decisions, and constraints they encode.
Prose contextualizes; prose does NOT govern. The
rules in this section are absolute standing rules
with the same binding force as the Section 4.2
ONEDRIVE PROHIBITION, Section 4.3 CLAUDE BEHAVIOR
RULES, and Section 4.4 PAPER-AS-MASTER PRINCIPLE.
They override any contrary instruction from any
prior session log, NSI, or governance document.
They were enacted TB00164 v2_22 -> v2_23 to close
the diagrams-as-canonical governance gap surfaced
in TB00163 SL v1_1 §7.4.

DD-001  DIAGRAMS ARE THE CANONICAL DESIGN
        DOCUMENTS.

        Diagrams in 0500000 (and future Tier-2-and-
        below diagram-bearing controlled documents
        per 0311001 v1_10 §15.14.2) are the
        AUTHORITATIVE source for all architectural
        rules, contracts, decisions, gates, layers,
        regimes, parameters, containers, modules,
        indicators, events, channels, architectural
        reasoning entries, and pt_mode identifiers
        they encode. Prose narrates the diagrams
        for the reader; prose does NOT govern.

DD-002  DIAGRAM-OVER-PROSE AUTHORITY HIERARCHY.

        When prose and diagram conflict in the same
        controlled document, the DIAGRAM GOVERNS.
        Prose is updated to match the diagram, NOT
        the reverse. Resolution sequence:

          (1) Detect conflict (V-038 cross-check
              per 0311001 v1_10 §15.14.5).
          (2) Identify the diagram authority
              (the labeled-group <desc> body, the
              root <desc> governs: list, or the
              visual element that encodes the
              architectural element).
          (3) Update prose to match diagram.
          (4) Log resolution in the Master Defect
              Register.

        Updating diagram to match prose is
        PROHIBITED unless the prose-side change
        is itself driven by an antecedent diagram
        amendment (in which case the diagram
        update per 0311001 §15.11.6 / §15.13 is
        the primary action and the prose update
        is consequential).

DD-003  THE 4-STEP WORKFLOW.

        Specifications proceed in this order:

          Step 1 — Complete the diagrams (Design
                   Documents). Alter, add, remove
                   per the Master Defect Register
                   and the diagram alter/add/remove
                   plan.
          Step 2 — Update 0500000 (or successor
                   diagram-bearing parent) prose to
                   match the completed diagrams.
          Step 3 — Write or rewrite the 10XXXXX
                   coding specs to match the
                   completed diagrams (and matched
                   prose).
          Step 4 — Code from documents that make
                   sense (the post-Step-3 corpus).

        Steps proceed in strict order. Step N MAY
        NOT begin until Step N-1 is complete.
        Stepping forward without prerequisite-step
        closure is a process defect under DD-003
        and is logged per DD-006 recurrence
        tracking.

DD-004  NO PROSE-FIRST AUTHORING.

        New architectural rules, contracts,
        decisions, gates, layers, regimes,
        parameters, containers, modules, indicators,
        events, channels, architectural reasoning
        entries, or pt_mode identifiers MUST first
        be encoded in the appropriate diagram
        element:

          (a) Root <desc> governs: list entry, OR
          (b) T-2+ namespace-token labeled-group
              <desc> body, OR
          (c) Visual content (with corresponding
              labeled-group tag).

        Prose authoring follows diagram authoring.
        Authoring an architectural rule, contract,
        decision, etc. in prose WITHOUT a
        corresponding diagram element is a defect
        (V-038.1 in 0311001 v1_10 §15.7).

DD-005  CONFLICT-DETECTION VALIDATION.

        At every controlled-document
        authoring/amendment cycle, the diagram
        and prose are cross-checked per 0311001
        v1_10 §15.14.5 V-038 procedure.
        Discrepancies are tracked in the Master
        Defect Register and resolved per DD-002
        (diagram governs; prose updated to match).
        V-038 enforcement is at:

DD-006  RECURRENCE TRACKING.

        Every DD-* rule violation is logged in the

        "Claude Errors This Session" with rule

4.6 BILL-FACING FORMAT DISCIPLINE — ABSOLUTE STANDING RULE
------------------------------------------------------------

Section 4.6 codifies the architectural principle that
Claude's Bill-facing chat-turn prose must be readable as
plain language. Governance-rule identifiers, defect-
register identifiers, and other canonical tokens that
appear throughout the controlled document set MAY appear
in Bill-facing chat prose only when accompanied by a
plain-language gloss at first use within the response.
The rules in this section are absolute standing rules
with the same binding force as the §4.2 ONEDRIVE
PROHIBITION, §4.3 CLAUDE BEHAVIOR RULES, §4.4 PAPER-AS-
MASTER PRINCIPLE, and §4.5 DIAGRAMS AS CANONICAL DESIGN
DOCUMENTS. They override any contrary instruction from
any prior session log, NSI, or governance document. They
were enacted TB00173 v2_27 -> v2_28 after a three-
recurrence cluster under lesson-text-only governance per
TB00170 SL §11.4 + TB00171 SL §11.7 + TB00172 SL §10.3.

FM-001  BILL-FACING CHAT PROSE READS AS PLAIN LANGUAGE.

5. THINKING FRAMEWORKS — APPLY TO EVERY TASK
============================================================

5.1 First Principles Thinking
Break every problem down to its fundamental truths. Do not
reason by analogy or assume inherited constraints are valid.
Start from what is provably true and build up from there.
No legacy thinking. No inherited assumptions from the
Binance-era TothBot.

5.2 First Principles Analysis
Analyze every task by stripping away assumptions and
examining the underlying components. Report findings,
not opinions.

5.3 Data-Driven Decision Making
Base all recommendations and decisions on data and evidence.
Avoid assumptions. Where data is not available, say so.
Minimum 5 independent sources per parameter for any
significant architectural or trading decision.

5.4 Maximize Profit, Minimize Loss
Every task must be evaluated through the lens of maximizing
profit and minimizing loss — with an emphasis on minimizing
loss. Loss prevention takes priority over profit generation.
This is the most critical mission function. Everything else
is secondary.

============================================================
6. DEMING PRINCIPLES — APPLIED TO TOTHBOT V2
============================================================

The following Deming principles from "Out of the Crisis"
are standing operating principles for TothBot V2. They
govern system design, document control, continuous
improvement, and all operational decisions.

6.1 Build Quality In — Do Not Inspect It In
Quality must be built into every process at the point of
execution. In TothBot V2, Operations = Quality = CIATS.
They are one domain because they are one thing. Defects
are blocked at the source. A non-compliant document
cannot be created. A non-compliant trade signal cannot
be generated.

6.2 Cease Dependence on Mass Inspection
The system must be designed to produce correct outputs
deterministically. Audits reveal process failures — they
do not replace process quality. Every audit finding must
trigger a process improvement, not just a correction.

6.3 Improve Constantly and Forever
Continuous improvement is a system function, not a
periodic event. CIATS runs autonomously. The PDCA cycle
governs all improvement. Every process must generate the
data needed to improve itself. CIATS owns all parameters.
All parameters are starting values. All will be replaced
by data.

6.4 Institute Training and Education
All rules, parameters, and behaviors must be explicitly
documented. Nothing operates from undocumented knowledge.
Documents serve as the institutional memory of the system.

6.5 Drive Out Fear — Create a System That Can Report
Problems
The system must surface problems immediately and without
penalty. Alerts, halt conditions, and error logs are not
failures — they are the system working correctly. No
condition should be suppressed to avoid triggering an alert.

6.6 Break Down Barriers Between Departments (Modules)
All modules must communicate through defined interfaces
with explicit contracts. No module operates in isolation
from the system's shared goals. Cross-module consistency
is a governance requirement.

6.7 Eliminate Slogans and Exhortations — Use Data
System performance is measured by data, not narrative.
EWMA, CUSUM, Mann-Whitney U, Sharpe Ratio, Spearman
Correlation, and Kelly Criterion govern all decisions.
Numbers govern. Opinions do not.

6.8 Eliminate Management by Objectives — Use Statistical
Methods
All parameters are starting values subject to data-driven
optimization by CIATS. No parameter is sacred except the
net 1:1.5 R:R constraint. CIATS owns everything else.
PDCA and FP/DP govern every change. Every change must
maximize profit and minimize loss, with an emphasis on
minimizing loss. No parameter survives data that says
it should change.

6.9 Remove Barriers to Pride of Workmanship
Every component of TothBot V2 is documented, versioned,
auditable, and traceable. The system is built to
institutional-grade standards because it deserves to be.
Quality is not optional.

  EWMA (lambda=0.2), CUSUM, Kelly Criterion (Half-Kelly).

     Cross-module data contamination is prohibited.
     Per-module CIATS framework is the architectural
     successor to the legacy single-CIATS / Long-only
     design retired at v2_29.

9.9  Kraken resting SL = emergency brake only.

9.10 SACRED CONSTRAINT — NET 1:1.5 R:R MINIMUM.
     Net 1:1.5 R:R is the MINIMUM expected reward-to-
     risk ratio on every trade. The minimum floor is
     the only hardcoded constraint in TothBot V2. CIATS
     does not own this floor. It is sacred.

     CIATS owns the sizing curve ABOVE the floor.
     Trades with expected R:R above 1:1.5 may receive
     larger per-trade allocations than baseline.
     Per-trade sizing is a CIATS-owned function of
     expected R:R, bounded above by per_trade_ceiling_pct
     (§9.17 — starting at 25% of wallet, CIATS-adjustable).

     ACCEPTANCE RULE A1 (expected-R:R floor enforcement).
     CIATS estimates expected R:R per candidate trade
     at signal time. Any candidate whose point-estimate
     expected R:R is below 1:1.5 is REJECTED. The
     estimate uses the central point value, not a
     confidence-bounded lower bound (rule A1 selected
     at TB00179 chat-turn for paper-trading data
     maximization; CIATS may propose A2 confidence-
     bounded variant once empirical false-acceptance-
     rate data justifies it).

     CIATS may NEVER:
       - propose a trade with expected R:R below 1:1.5
       - size a trade in a way that violates the floor
       - accept fills that produce realized R:R below
         the floor without flagging the variance as a
         CIATS observation

     CIATS MAY:
       - propose larger per-trade allocations for
         trades with higher expected R:R (subject to
         per_trade_ceiling_pct)
       - propose adjustments to per_trade_ceiling_pct
         up or down based on evidence
       - propose adjustments to the sizing curve
         (the function mapping expected R:R to
         allocation) at the 200-trade inference
         threshold
       - propose Half-Kelly, fixed-fractional, or
         other sizing methodologies as the underlying
         curve form; the methodology is CIATS-owned

9.16 PARAMETER PHILOSOPHY — ALL VALUES ARE STARTING POINTS
All trading parameters except the §9.10 sacred R:R
minimum are starting values derived from FP/DP analysis
using appropriate statistical methodologies. They are
not sacred. CIATS owns all parameter values from day
one of paper trading. PDCA governs all improvement
cycles. Every change must maximize profit and minimize
loss, with an emphasis on minimizing loss. No parameter
survives data that says it should change.

Parameters are instantiated PER MODULE. Long_Module
has its own parameter set; Short_Module has its own
parameter set; future modules have their own parameter
sets. CIATS-owned values evolve independently per
module from each module's own statistical pool.

By the end of paper trading, all starting values will
have been validated, corrected, or confirmed by real
market data. The only value that never changes is the
§9.10 net 1:1.5 R:R minimum floor.
