2.2 Root Cause Analysis — DEFECT-SS-009
------------------------------------------------------------

DEFECT-SS-009 (CRITICAL — CONFIRMED AND FIXED)

Root cause: _regime_wrapper in startup_sequence.py called
  rge.run_daily_computation() with no universe argument.
  run_daily_computation() uses self._universe when no
  universe is passed. rge._universe is set at RegimeEngine
  construction time from _load_universe(config).
  _load_universe() returns ["BTC/USD"] only (TOTHBOT_UNIVERSE
  env var not set; default is "BTC/USD").

  The full 50-pair monitored universe is built by WSManager
  at runtime from the Kraken instrument snapshot — it is NOT
  available at construction time. rge._universe = ["BTC/USD"]
  for the lifetime of the process.

  Result: regime computed for 1 pair (BTC/USD) only.
  All 49 other monitored pairs had NO_REGIME_DATA.
  Gate 3 blocked all non-BTC entries for the entire session.
  This also affects the daily 00:00 UTC regime refresh —
  same wrapper, same defect.

  Evidence:
    REGIME_STARTUP_SEED_COMPLETE in ~1.25s (BTC/USD: 1.1s
    stagger + ~0.15s REST fetch = ~1.25s correct for 1 pair).
    All non-BTC pairs showing GATE_3_FAIL NO_REGIME_DATA
    27 minutes after SYSTEM_OPERATIONAL.

Fix: _regime_wrapper now passes list(wm.monitored_universe)
  to rge.run_daily_computation(universe=universe). Python
  closures capture wm by reference — at call time,
  wm.monitored_universe contains the full ranked 50-pair
  universe. Guard: if wm.monitored_universe else None.

2.7 DEFECT-WM-RECONNECT-001 — NEW DEFECT (CRITICAL)
------------------------------------------------------------

Reclassification: OI-016 DEFECT-WM-CRASH-001 is now
closed as "diagnosed." The actual production defect is
reconnect resilience, not crash detection.

DEFECT-WM-RECONNECT-001 (CRITICAL — diagnosed, NOT fixed)

Component:   ws_manager.py dv1_19
Governing:   1011002_WS_Manager_Coding_Spec dv1_18

Problem:
  _ping_loop (lines 2353, 2360), _recv_loop_public (line
  975), _send_public (line 735), and _initiate_reconnect
  (line 3241) propagate routine WebSocket disconnects out
  of the asyncio TaskGroup. Any disconnect triggered by
  CloudFlare proxy cycling or transient network conditions
  terminates the entire TothBot process.

Required behavior:
  ConnectionClosedError, ConnectionClosedOK, and
  ConnectionResetError within the public WebSocket loops
  must be caught locally and trigger orderly reconnect
  without process termination. Only truly unrecoverable
  conditions (e.g., repeated reconnect failure across N
  attempts) should propagate to WS_MGR_FATAL.

_initiate_reconnect at line 3241 currently raises
RuntimeError("Fatal reconnect failure — systemd restart
required") — this logic needs review. The reconnect path
should retry with backoff, not immediately fatal-exit
after one failure.

Impact:
  System experiences ~1-2 crashes per hour (mean interval
  1h 37m over 13h window). Each crash loses in-flight
  WebSocket state, forces 60s+ regime re-seed, blocks
  Gate 3 during re-seed, and creates observational noise
  across all downstream components. CIATS data quality
  degraded by constant restarts.

Severity: CRITICAL. Blocks stable paper trading.

Fix scope (NEXT SESSION — NO PATCHES):
  ws_manager.py dv1_19 → dv1_20 (full file, full DC)
  1011002 WS_Manager_Coding_Spec — review reconnect rules;
    increment version if new rules added
  1010000 Trading_System_Coding_Anchor — registry update
