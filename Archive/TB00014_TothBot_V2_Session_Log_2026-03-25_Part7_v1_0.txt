SEARCH 6 — asyncio semaphore max concurrent positions
------------------------------------------------------------

Sources: rednafi.com asyncio semaphore article, Soumendrak
semaphore blog, superfastpython asyncio gather concurrency,
Python 3.14 asyncio docs (full fetch), copyprogramming
concurrency guide, dev.to semaphore guide, death.andgravity
limit concurrency, GitHub benfasoli gist, snok/self-limiters.
References: 9 additional unique.

FINDINGS:
S6-F1: ★ NEW-10 — asyncio.BoundedSemaphore is confirmed
       as the correct Python primitive for limiting concurrent
       position count. BoundedSemaphore raises ValueError on
       over-release (release() called more than acquire()),
       which is safer than regular Semaphore for position
       tracking (over-release = bug indicator).

S6-F2: CONFIRMED — Python 3.11+ introduced TaskGroup for
       structured concurrency. asyncio.Semaphore remains
       the correct primitive for concurrency limiting.
       TaskGroup is for managing groups of related tasks
       with automatic cancellation on error.

S6-F3: CONFIRMED — Check/acquire separation is the correct
       asyncio pattern for TOCTOU safety:
         if not semaphore.locked():  # non-blocking check
             await semaphore.acquire()  # actual acquire
       This matches our Gate 7 / Execution Engine separation.

S6-F4: CONFIRMED — Python official asyncio.Semaphore docs:
       "The preferred way to use a Semaphore is an async
       with statement." However, for position tracking
       where acquire and release occur at different code
       points, explicit acquire/release is required.

SEARCH 6 CONCLUSION: asyncio.BoundedSemaphore confirmed
as correct implementation. Full pattern documented.
