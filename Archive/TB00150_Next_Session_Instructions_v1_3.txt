1.2 STREAM A2 (TB00152) — 1011004 NEW dv1_0 SCOPE
------------------------------------------------------------

Per 0500000 dv1_23 §12.8 compliance mapping (which
codifies 1011004 NEW dv1_0 scope), the spec must
specify:

  (a) Module purpose and responsibilities.
      Execution Engine is the orchestration module
      between Signal Pipeline (Gate 8 output) and
      WSManager (dispatch seam). EE maintains entry
      order registry, processes executions channel
      callbacks, dispatches Phase 1 entry orders and
      Phase 2 batch_add (TP + emergSL atomic).

  (b) HR-EE-013 paper-live behavioral parity.
      Execution Engine has NO mode-conditional
      branching. Mode (paper / live) is enforced
      upstream at WSManager dispatch seam (0500000
      dv1_23 §12.3). EE code path is identical in
      either mode. Caller contract per 0500000
      dv1_23 §12.3.

  (c) CIATS-owned parameters (read via params.get()).
      entry_timeout_sec        : GTD parameter for
                                 entry orders (30-60s
                                 starting; CIATS-
                                 owned).
      per_trade_pct            : Sizing parameter
                                 (5% starting;
                                 transitions to
                                 Half-Kelly at 200
                                 trades; CIATS-
                                 owned).
      max_concurrent           : Concurrency limit
                                 (20 starting;
                                 CIATS-owned).
      All read at evaluation time, not cached.

  (d) Inputs.
      From Signal Pipeline (Gate 8 output):
        symbol, side (LONG only — Long Module),
        entry_price (limit), tp_price, emerg_sl_price,
        size_usd, regime_metadata.
      From WSManager (executions channel callbacks):
        execution events (filled, partially_filled,
        canceled, expired) with sequence tracking
        per A-10.

  (e) Outputs.
      To WSManager seam:
        add_order calls (Phase 1: limit entry GTD)
        batch_add calls (Phase 2: TP + emergSL atomic)
        cancel_order calls (entry timeout)
      To Position Mirror:
        position state updates (PT-MODE-001 paper /
        live identical schema)
      To Logger:
        ENTRY_SUBMITTED, ENTRY_FILLED,
        ENTRY_TIMEOUT_CANCEL, TP_PLACED,
        EMERGSL_PLACED events
      To CIATS:
        Trade Outcome Bus events (via Logger;
        CIATS reads from log files only).

  (f) Hard rules (must / must not).
      MUST: All dispatches go through WSManager seam
            methods. NO direct outbound to Kraken.
      MUST NOT: Branch on paper_trading_mode. EE is
                mode-blind per HR-EE-013.
      MUST: Read CIATS-owned parameters at evaluation
            time via params.get().
      MUST NOT: Cache parameter values across
                pipeline cycles.
      MUST: Cancel un-filled entry orders at
            entry_timeout_sec elapsed.
      MUST: Submit TP + emergSL atomically via
            batch_add after entry fill.
      MUST: 1:1.5 net R:R enforcement at order-
            placement time (HARDCODED — sacred).
      MUST: Cap max_concurrent positions before
            new entry submission.

  (g) Triggers and downstream triggers.
      Triggered by: Signal Pipeline Gate 8 output
        (5-minute candle close cycle).
      Triggers: WSManager seam method calls;
                Position Mirror state update;
                Logger events; (downstream) CIATS
                Trade Outcome Bus consumption.

  (h) Logging.
      Every state mutation logs an event to Logger.
      Event taxonomy: ENTRY_SUBMITTED,
        ENTRY_FILLED, ENTRY_PARTIAL_FILL,
        ENTRY_TIMEOUT_CANCEL, ENTRY_REJECTED,
        TP_PLACED, EMERGSL_PLACED, TP_FILLED
        (delegated to Exit Controller),
        EMERGSL_FILLED (delegated to Exit
        Controller), CONCURRENCY_LIMIT_HIT.

  (i) Error handling and reconciliation.
      Sequence gap detection per A-10: WSManager
      detects executions channel sequence gap and
      triggers REST GetOpenOrders reconciliation.
      EE consumes reconciliation result through
      WSManager and reconciles its entry order
      registry.

  (j) Testing requirements.
      Unit tests per 1021001 Unit_Test_Specification
      dv1_0.
      Integration tests per 1021002
      Integration_Test_Specification dv1_0.
      Paper trading tests per 1021003
      Paper_Trading_Test_Specification dv1_0.
