6.3 STEP C — FP/DP analysis for OI-028
------------------------------------------------------------

Candidate root causes to evaluate (not exhaustive):

  (a) ConnectionClosedError not in WM-RECONNECT-016
      transient types → raises through TaskGroup →
      _initiate_reconnect fires → MAX_RECONNECT_ATTEMPTS
      failure (the test specifically increased to 20
      in OI-020, so this is unlikely unless 20 also
      insufficient — but MAX_RECONNECT_EXCEEDED = 0
      in the sweep, so this is NOT the root cause).

  (b) _initiate_reconnect's retry schedule EXHAUSTS
      (the 10/20 attempts) despite network being fine.
      BUT: MAX_RECONNECT_EXCEEDED = 0 — so this is
      also NOT what fired. Need to understand why
      _initiate_reconnect raised RuntimeError without
      incrementing the EXCEEDED counter.

  (c) Re-entrancy defect: concurrent disconnect
      detection by both recv loops. WM-RECONNECT-018
      specifies _reconnect_done_event and
      _fatal_reconnect_failure flag to collapse dual
      cycles. Check whether these guards were
      exercised correctly — or absent.

  (d) "no close frame received or sent" is the library-
      level symptom. This is NOT the same as a normal
      ConnectionClosedOK or ConnectionClosedError. It
      may mean the websockets library is raising a
      variant that _recv_loop_public's except-arm
      does not catch, and which is therefore
      propagating up as an unhandled TaskGroup sub-
      exception.

Verdict for TB00112: hypothesis (d) is first-principles
most likely given the library-version-dependent wording
and the 1011002 explicit enumeration of transient
types. Diagnose by reading both 1011002 transient-type
enumeration AND actual except-clauses in
_recv_loop_public.

NO FIX WRITTEN until FP/DP analysis confirms root
cause AND Bill approves the direction.

6.4 STEP D — FP/DP analysis for OI-029
------------------------------------------------------------

Read logger.py SMTP send path. Two specific questions:

  (d1) What exception types are caught from aiosmtplib?
       Is aiosmtplib.errors.SMTPConnectError caught?
       Is asyncio.TimeoutError caught? Is the
       underlying ConnectionError caught?

  (d2) On catch, does the logger retry? With what
       backoff? Or is failure terminal after a single
       attempt?

Candidate fix:
  Wrap send in a 3-attempt retry with exponential
  backoff (1s, 2s, 4s) for transport-level errors.
  Preserve immediate failure for permanent errors
  (authentication failure, 550 recipient rejection)
  to avoid spurious retries on real problems.

Full DC package required: logger.py version bump +
1011007 Logger Coding Spec version bump + possibly
0511006 Logger Specification update if the retry
behavior is considered architectural.
