      Multiple instances at 00:18:32, 00:18:39, 00:18:41,
      00:18:46, 00:18:52, 00:19:01.
    - uvloop ConnectionResetError tracebacks at 00:14:52
      (SSL handshake-complete failures during reconnect).
    - Recursive failure mode: each reconnect re-issued
      the full expanded-universe subscribe sequence ->
      Kraken rate-limit rejection -> disconnect -> repeat.
      Each rate-limit-triggered reconnect generated
      RECONNECT_INITIATED alerts -> alerts dispatched ->
      ALERT_SENT records logged -> logger queue overflow ->
      records dropped.

  ROOT CAUSE (FP/DP):
    Kraken WS v2 enforces a per-connection message rate
    limit (per Kraken docs: "varies depending upon the
    load on the system"; published threshold not
    available). dv1_24 expanded the monitored universe
    from ~50 (dv1_23 Top-50 cap + USD/USDC only) to ~200+
    (USD + USDC + USDT, no cap). ws_manager.py issues
    subscribe messages at the rate produced by the
    initialization loop; for ~200+ pairs, this exceeded
    Kraken's dynamic threshold and triggered the
    rejection cascade.

  PRODUCTION-IMPACT ASSESSMENT:
    - Trading logic effectively offline (no clean WS
      connection to receive 5-min candle close events).
    - CIATS data integrity compromised (logger queue
      overflow dropped INFO-level ALERT_SENT records and
      DEBUG-level LIQUIDITY_SEEDED_DETAIL records).
    - HIGH-level alert emails were emitted for each
      reconnect but the corresponding ALERT_SENT log
      records were dropped, creating evidence gaps.
    - Paper-to-live parity violated (production-grade
      stability is the D-06 requirement; system was not
      stable).

  DECISION: emergency rollback to dv1_23.

3.10 ROOT CAUSE INVESTIGATION + ARCHITECTURAL THEORY

Bill identified the Top-N=50 cap in dv1_23 may have
served as a de facto Kraken WS subscribe rate-limit
protective mechanism, even though no spec document
recorded that rationale. FP/DP analysis confirmed
this hypothesis:

  EVIDENCE:
    - 0711003 dv1_3 §AS-002 retirement rationale states
      verbatim: "The prior monitored_universe_n = 50 cap
      was never FP/DP-validated. It was asserted in
      0500000 AR-070 rationale without dedicated
      research. TB00128 Ratified Decision D-08 retires
      the cap."
    - D-08 was made on the basis of "no documented
      rationale -> remove" without considering empirical
      operational constraints.
    - dv1_23 ran 20+ hours stable with 50 subscribed
      pairs and zero rate-limit errors. dv1_24 with
      ~200+ pairs hit rate-limit immediately.

  GOVERNANCE LESSON (Chesterton's Fence pattern):
    Removing a constraint without investigating WHY the
    constraint was originally introduced is unsafe.
    "No documented rationale" is NOT equivalent to "no
    operational rationale."

Bill's secondary hypothesis (theory under investigation):
the Kraken msg rate limit will not be a problem in
STEADY STATE — only during startup and reconnect bursts.
FP/DP analysis supported this:

  - Kraken msg rate counter measures CLIENT-OUTBOUND
    messages (subscribe, addOrder, cancelOrder, ping).
    Server-outbound messages (data, snapshots,
    heartbeats Kraken pushes TO us) are not counted
    against client rate limit.
  - Steady-state outbound message rate for TothBot is
    extremely low: ~1 ping/30sec, ~20 add/cancel/day,
    heartbeat replies. Well below any plausible
    threshold.
  - dv1_23 empirically confirmed steady-state rate-
    limit safety at 50 pairs. Theory predicts same at
    200+ pairs.

Bill authorized clean-slate ws_manager.py rewrite if
needed. The original dv1_22-and-earlier code base is
considered tainted by undocumented Claude-introduced
universe constraints; the Top-N cap was the most
visible but may not be the only one. Clean-slate
rewrite is permitted; "build a better mousetrap" using
the lessons learned.

Bill chose Option (a): proceed with empirical probe
script to verify the steady-state-safe theory before
committing to a specific architectural path.

3.11 INVESTIGATION SUMMARY — FOUR ARCHITECTURAL PATHS

Three sequential web searches against Kraken docs
established the architectural option space:

  PATH 1  Batched subscribe (single subscribe message
          per channel containing all pairs in the
          symbol[] array). Eliminates max_concurrent
          dependency. Low complexity. STRONGEST
          candidate.

  PATH 2  Multiple WS connections, batched subscribes
          per connection. Fallback if PATH 1 hits an
          undocumented per-message symbol-count cap.

  PATH 3  Tiered HOT/WARM (subscribe only active-
          position pairs). RETIRED. Adds complexity for
          no benefit when batched subscribes decouple
          subscription count from position count.

  PATH 4  PATH 1 + adaptive backoff on "Exceeded msg
          rate" rejections. Pairs with PATH 1 or PATH 2
          for resilience.

Architectural decision deferred to TB00141+ pending
empirical probe-script results.
