3.0 THE DUAL-OWNERSHIP SPLIT (THE PARADIGM CHANGE - RATIFIED
    BY BILL THIS SESSION; THIS IS THE ROOT OF THE CONFUSION).

    Ownership is NO LONGER one-order / one-module. A single
    entry order, once partially filled, BISECTS into two
    concurrently-owned objects governed by two different
    modules at the same time. The boundary is the fill
    itself.

    Worked example (Bill's ratified model):
      - A $10 entry order is opened. While WORKING (unfilled)
        it is owned by the SELECTION module (entry-side
        control).
      - It partially fills for $5. The single order now
        SPLITS into two objects:
          * The $5 FILLED = a real position. Ownership
            TRANSFERS to the EXIT Criteria / Exit Controller.
          * The $5 UNFILLED = the remaining working order.
            Ownership STAYS with the SELECTION module.
      - If the SIGNAL_DECAY entry-cancel fires BEFORE the
        remaining $5 fills, the SELECTION module is STILL IN
        CONTROL of the unfilled portion and the SELECTION
        module cancels that unfilled $5.
      - The Exit Controller is NOT the canceller. It owns
        and manages only the $5 that already filled.

    Net rule: filled qty -> Exit Controller; unfilled qty ->
    Selection module. The cancel of the unfilled remainder
    on SIGNAL_DECAY is a SELECTION-MODULE (entry-criteria)
    action, NOT an Exit-module / AR-054 action.

    Registry consequence (AR-053 Pending Order Registry must
    reconcile across the split): the cancelled unfilled qty
    is cleared from the pending registry; the filled qty is
    registered as an open position.

This is EXACTLY DEC-065 Q2. It is RATIFIED. It is not to be
re-opened or re-asked. It splits into two halves:

3.1 HALF A - THE ALREADY-FILLED PORTION (EXIT-SIDE).
    Owned by the Exit Controller. Its dedicated SAFETY NET
    is ar:AR-054 (entry order partial fill + expiry/cancel,
    direction-symmetric protection): when a GTD post_only
    entry order partial-fills and the remainder
    cancels/expires (exec_type=expired OR canceled with
    cum_qty>0) leaving a real unprotected stub with NO TP and
    NO emergSL, AR-054 dispatches market SELL (Long_Module) /
    market BUY reduce_only=true (Short_Module) to reduce the
    stub to zero.
    SCOPE NOTE: AR-054 is the Exit-side safety net for the
    FILLED stub. AR-054 is NOT the mechanism that cancels the
    unfilled remainder on SIGNAL_DECAY - that is the
    Selection-module action in 3.0 / 3.2.
    STATUS: ALREADY CANONICAL. AR-054 is already in D2's root
    governs list (live file line 2175) and defined at
    0500000 sec 4 lines 5169-5195. Companion ar:AR-053
    (per-module Pending Order Registry, capital-commitment
    tracking, sec 4 line 5122) is also already in D2 governs.
    NOTHING TO AUTHOR for Half A. Done.

3.2 HALF B - CANCEL THE UNFILLED REMAINDER ON SIGNAL
    DETERIORATION (the SIGNAL_DECAY entry-cancel trigger).
    Per DEC-065 Q2: if the entry signal (SSS) deteriorates
    below threshold while the entry order is still working
    (unfilled / partially filled), the SELECTION module -
    which still owns the unfilled remainder per 3.0 - cancels
    the unfilled / remaining qty; the Exit Controller owns
    any already-filled portion (3.1).
    OWNERSHIP OF THE CANCEL ACTION: SELECTION module
    (mod:Selection_Controller), NOT the Exit Controller and
    NOT AR-054. The mechanical cancel dispatch to the
    exchange may route through the Execution Engine, but the
    control/decision to cancel the unfilled remainder sits
    with the Selection module.
    DEC-065 Q2 explicitly states (TB00442 SL sec 2.4
    confirmed) that this trigger mechanism DOES NOT EXIST in
    the current entry criteria - it is NET-NEW DESIGN to be
    authored (detection condition into 0711001
    SSS_Signal_Model; cancel-action ownership = Selection
    module). Tracked as CF-TB00442-A. Sequenced as 0911002
    sec 5A Track B, separate from the Track A reconciliation
    cascade.
    STATUS: RATIFIED-AS-A-DECISION, NOT YET AUTHORED into any
    diagram or spec. This is the open work.

3.3 THE CIRCLE, NAMED.
    Each prior pass conflated FOUR distinct things:
      (i)   the exit_reason ENUM (Track A, D3/D5/D6) -
            SIGNAL_DECAY removed from it (Q2); D2 has no
            enum; RA-6 D2 is empty by decision.
      (ii)  the already-filled-portion safety net (AR-054,
            Exit-side) - already canonical, already in D2.
      (iii) the unfilled-remainder cancel trigger (Track B,
            CF-TB00442-A), owned by the SELECTION module -
            ratified but net-new, not yet authored.
      (iv)  the DUAL-OWNERSHIP SPLIT itself (3.0) - one
            partial fill bisects one order into a
            Selection-owned unfilled remainder + an
            Exit-owned filled portion, concurrently.
    Conflating (i)/(ii)/(iii)/(iv) - and specifically
    mis-assigning (iii) to the Exit side instead of the
    Selection module - is what kept re-surfacing. They are
    now separated explicitly. (i) and (ii) are CLOSED. (iii)
    remains and is deferred to the next session(s), now with
    correct Selection-module ownership. (iv) is the framing
    that resolves the recurring confusion.
