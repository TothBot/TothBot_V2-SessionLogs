2.4 Mid-session discovery — OI-023 data loss defect
------------------------------------------------------------

During observation-window idle time, Bill explored what
productive work could run during the wait. Conversation
led to scoping an Operator Observer Dashboard to give Bill
"is it running?" and "is it making money?" at-a-glance
views. Design inquiry for the P&L view (time periods,
tax-year tracking, CSV export) surfaced a concern: how is
trade data preserved if logs rotate?

Investigation of the Logger specification and deployed
code confirmed a CRITICAL defect:

  TRADE_CLOSE records (the authoritative record of every
  closed position — entry/exit prices, fees, P&L, regime
  tags, signal params, timestamps) are written ONLY to
  tothbot.log. No separate permanent store exists.

  tothbot.log is managed by RotatingFileHandler. Deployed
  config retains only a small number of rotated files (4
  files observed on VPS: tothbot.log + tothbot.log.1/2/3).
  At ~50 MB/file and current write rate, this yields
  approximately 6 days of on-disk history.

  Consequence: trade records more than ~6 days old are
  PERMANENTLY LOST. This breaks:
    - CIATS 200-trade statistical floor (rolling corpus
      truncated to ~6 days of trades)
    - IRS Form 8949 / Schedule D recordkeeping (3-year
      minimum retention required by 26 CFR 1.6001-1)
    - Audit trail for any post-incident forensic work
    - Annual / YTD / 365d P&L for operator dashboard

  Filed as OI-023 DEFECT-TRADE-RECORD-001 (full write-up
  Section 3.6 below).

3.6 OI-023 DEFECT-TRADE-RECORD-001 — CRITICAL — NEW

    Title: Trade records lost to log rotation within ~6
    days of closure.

    Severity: CRITICAL.

    Affected scope:
      - logger.py (current deployed: dv1_3)
      - 1011007 Logger_Coding_Spec (current: dv1_3)
      - 0511006 Logger_Specification (current: dv1_0)
      - Live tothbot.log trade records already older
        than ~6 days are UNRECOVERABLE.

    Root cause: TRADE_CLOSE events are emitted via the
    same Logger path as every other event, landing in
    tothbot.log. tothbot.log is managed by
    RotatingFileHandler with limited retention. No
    independent permanent trade store exists.

    Observable symptom: VPS ls of /home/tothbot/logs/
    shows tothbot.log.1/2/3 dated Apr 20, Apr 18, Apr 15
    — oldest retained history is ~6 days. Any trade
    closed before that is gone.

    Blast radius:
      - CIATS cannot build a 200-trade statistical corpus
        if paper-trade throughput is below ~33 trades/day
        (200 trades / 6 days). Below that threshold,
        CIATS activation is structurally blocked by data
        loss, not by trade volume.
      - US tax filing: 26 CFR 1.6001-1 requires retention
        of records supporting tax return items for 3
        years minimum (longer in audit-extension cases).
        Dashboard-derived P&L without source records is
        not audit-defensible.
      - Post-incident forensics: any investigation of a
        historical trade anomaly is impossible once
        records age out.

    Proposed fix (high level — full design in TB00108):
      - Add append-only permanent trade record file:
        /home/tothbot/records/trades_<YYYY>.jsonl
        (annual-segmented, no rotation, one JSON per
        line, one line per TRADE_CLOSE event).
      - Logger writes TRADE_CLOSE events to BOTH the
        main rotating log (diagnostics) AND the annual
        trade record file (permanence). Single source
        of truth remains the Logger's TRADE_CLOSE record
        contract — no duplicate computation.
      - On-demand CSV export transformation (for tax
        tools and accountants) derived from the annual
        JSONL. No dual write paths.

    Fix governance:
      - logger.py dv1_3 -> dv1_4
      - 1011007 Logger_Coding_Spec dv1_3 -> dv1_4
        (new section specifying annual trade record
        file, path, contents, write contract, retention)
      - 0511006 Logger_Specification — evaluate whether
        architectural update is needed; likely yes
        (two-stream record architecture)
      - 1010000 Trading_System_Coding_Anchor registry
        bump

    FP/DP decisions locked for TB00108:
      Location: /home/tothbot/records/trades_<YYYY>.jsonl
      Contents: TRADE_CLOSE records only
      Format:   JSONL primary; CSV generated on demand
      Retention:Annual file; no rotation; permanent
      Timing:   Deploy immediately in TB00108 bundle
