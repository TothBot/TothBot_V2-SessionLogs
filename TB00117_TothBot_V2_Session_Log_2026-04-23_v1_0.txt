2.6 OI-029 diagnosis (LOCKED)
------------------------------------------------------------

ROOT CAUSE:

logger.py line 166:
  ALERT_SMTP_TIMEOUT_SEC: int = 5   # fire-and-forget

This 5-second timeout applies to the full SMTP
conversation (logger.py lines 296-301):

  with smtplib.SMTP(host, port, timeout=5) as server:
      server.starttls()
      server.login(user, pass)
      server.sendmail(user, [ALERT_EMAIL], msg.as_string())

Four sequential network round-trips (connect, starttls,
login, sendmail) share a single 5-second read/write
budget per socket operation. Any single operation that
takes >5 sec to respond triggers:

  "Connection unexpectedly closed: The read operation
   timed out"

This is exactly the failure mode observed on the VPS:

  Apr 22 14:35:21 HIGH RECONNECT_INITIATED alert
                  -> SMTP FAILED (read timeout)
  Apr 22 14:57:29 HIGH RECONNECT_TRIGGERED alert
                  -> SMTP FAILED (read timeout)

except block at line 317 catches ALL SMTP exceptions,
writes stderr + emits ALERT_SMTP_FAILED. NO RETRY
logic. NO BACKOFF. One failed send = one permanent
alert loss (except for the stderr fallback which
appears in journalctl).

logger.py line 391 docstring:
  "NEVER on the hot path. SMTP blocking here is
   acceptable."

The function is off the hot path. Retry-with-backoff is
safe to add.

FIX VECTOR (to deliver in TB00118 DC package):

  Change 1: ALERT_SMTP_TIMEOUT_SEC: 5 -> 20
            (covers transient server load / slow TLS
            handshake without blocking excessively)

  Change 2: Wrap SMTP conversation in a 2-attempt loop
            with 1-second backoff between attempts.
            Total worst-case wall time: 20 + 1 + 20 = 41s.
            Still off hot path, still bounded.

  Change 3: Only emit ALERT_SMTP_FAILED if BOTH attempts
            fail (single-transient retry absorbs 99% of
            read-timeout cases).
