DEC-001  Master Defect Register publication scope
------------------------------------------------------------

STATUS: OPEN

QUESTION:
Does TB00153_Master_Defect_Register_v1_0.txt become a
controlled document committed to TothBot_V2-Docs repo
with its own DCN lifecycle (and therefore subject to
DCN versioning when defects close), or does it remain
a one-shot session deliverable carried only inside the
TB00153 session-log archive?

FIRST-PRINCIPLES FRAMING:
The register must be referenced from every TB00154+
diagram-rework session NSI to identify which defects
the session closes. Two paths satisfy that need:

  Path A — Controlled doc in TothBot_V2-Docs.
    Register lives at a stable repo path with version
    suffix (v1_0). When defects close, register is
    re-published as v1_1, v1_2, etc., with a defect-
    closure log section. NSIs reference the register
    by DCN + latest version.

  Path B — Session-archive-only.
    Register lives only in TBXXXXX_Session_Logs\ as a
    TB00153 deliverable. TB00154+ NSIs reference it
    by file path. As defects close, status is tracked
    in the closing session's SL — no register re-
    publish.

CLAUDE RECOMMENDATION: Path A (controlled doc).

DATA JUSTIFICATION:
  - 40 defects across 7 diagrams, 8 stages
    (TB00154 - TB00161). Status will mutate session-
    by-session for ~8 sessions minimum.
  - Without a single source-of-truth document that
    versions forward, defect-closure status will
    fragment across 8+ session logs. Re-deriving
    "which defects remain open" before each new
    session becomes a recurring cold-load cost.
  - Pattern matches TB00000 (governance) and 0311001
    (DCN standards) — both controlled docs that
    version forward as governance evolves.
  - First-principles loss-minimization: stale defect
    state = repeat work or missed gates. Controlled
    doc with explicit version cadence eliminates that
    failure mode.

ALTERNATIVE PATH B TRADE-OFFS:
  - Lower governance overhead (no re-publish per
    closure)
  - But: defect status discovery becomes O(N sessions)
    instead of O(1 doc fetch)
  - And: no canonical "open defects remaining" view
    available to TB00154+ pre-flight

BILL ACTION REQUIRED:
Ratify Path A or Path B. If Path A: authorize commit
of TB00153_Master_Defect_Register_v1_0.txt to
TothBot_V2-Docs at path
  /Architecture/Master_Defect_Register/
  TB00153_Master_Defect_Register_v1_0.txt
and authorize re-publication cadence (one re-publish
per stage close, i.e., v1_1 after Stage A close,
v1_2 after Stage B close, etc.).

DOWNSTREAM IMPACT IF DEFERRED:
TB00154 NSI must still be authored. If DEC-001 is
unresolved at TB00153 close, TB00154 NSI will
reference the register by session-archive path
provisionally; if Bill subsequently chooses Path A,
references throughout TB00154+ NSIs will require
patch. Recommend ratification at TB00153 close.

DEC-002  D4 Regime Taxonomy count reconciliation
------------------------------------------------------------

STATUS: OPEN

QUESTION:
D4 (Image4_T2_R1_Regime_Taxonomy) currently asserts a
6-regime taxonomy in 0500000 dv1_24 §6 prose
(REGIME_TRENDING_UP, REGIME_TRENDING_DOWN,
REGIME_RANGING_TIGHT, REGIME_RANGING_WIDE,
REGIME_VOLATILE, REGIME_CHOP). Coding spec 1011007
Regime_Detection_Coding_Spec dv1_X (per TB00148
audit cluster 4) describes a 5-regime classifier
output (omits REGIME_CHOP per the spec authored
during paper-trading dev sprint).

This is paper-as-master ambiguous: which is the
canonical truth — the prose+diagram pair (6) or the
coding spec (5)?

FIRST-PRINCIPLES FRAMING:
PA-001 says paper code is master. But paper code
output has been observed (per TB00148) to emit only
5 regime labels in production logs across the entire
paper-trading window. Therefore:

  - Empirical paper-code behaviour: 5 regimes.
  - 0500000 prose + diagram: 6 regimes.
  - 1011007 spec: 5 regimes.

By PA-001, paper code is master → 5 regimes is
canonical. 0500000 prose + diagram are WRONG.

But: was REGIME_CHOP a deliberate retirement, or a
silent drift? TB00115 - TB00150 contains no decision
record retiring REGIME_CHOP. This is a Chesterton's-
fence open item per RR-SCOPE-009.

CLAUDE RECOMMENDATION:
  Step 1 — Ratify 5-regime canonical (matches paper
    code, matches 1011007 spec).
  Step 2 — Document REGIME_CHOP retirement as a
    paper-era silent decision; record retroactively
    in this register and in 0500000 prose §6 amend.
  Step 3 — D4 diagram authored with 5 regimes; D4
    SVG includes a NOTE band acknowledging
    REGIME_CHOP was historically considered and
    deprecated.

DATA JUSTIFICATION:
  - PA-001 paper code = master is unambiguous.
  - Paper-code output history: 5 regimes only.
  - 1011007 spec already at 5 regimes.
  - Cost to break tie toward 6 (re-introduce
    REGIME_CHOP into paper code): code amendment
    + spec amendment + new test corpus + new
    classifier training. Massive cost vs. ratifying
    the empirical state.

ALTERNATIVE: Ratify 6-regime as canonical.
  Trade-off: PA-001 violation. Requires Bill written
  authorization per PA-006. Forces paper-code
  amendment (out-of-scope this session per RR-SCOPE-
  002). Forces re-test of regime classifier.
  NOT RECOMMENDED.

BILL ACTION REQUIRED:
Ratify 5-regime canonical, with REGIME_CHOP
retirement documented retroactively in TB00153 SL
appendix and amended into 0500000 §6 during Stage H
(D4 rework session).

DOWNSTREAM IMPACT IF DEFERRED:
Stage H (D4 rework) is BLOCKED until DEC-002
ratifies. Stage H is the LAST stage in the 8-stage
plan, so deferral does not block earlier stages —
but DEC-002 must be resolved before TB00161
(Stage H) is scheduled. Recommend resolution at
TB00153 close to remove TB00161 pre-flight risk.

DEC-003  Stage cadence — one diagram per session vs
         bundled multi-diagram sessions
------------------------------------------------------------

STATUS: OPEN

QUESTION:
The Diagram Alter/Add/Remove Plan defines 8 stages
(Stage A through Stage H) covering Tier 1 Context +
D1-D7. Should each stage be exactly one chat session
(8 sessions: TB00154 Stage A, TB00155 Stage B, ...,
TB00161 Stage H), OR should certain low-defect stages
be bundled (e.g., D5/D6 in one session because both
require T-2+ tagging only and have no architectural
content changes)?

FIRST-PRINCIPLES FRAMING:
Trade-off axis: session-count economy vs. error-
budget per session.

  Path A — One diagram per session (8 sessions).
    Each session has narrow scope, focused defect-
    closure, low context-window pressure, smaller
    blast radius if a regression slips. NSI / SL
    cadence stays predictable (1 diagram = 1 SL =
    1 NSI).

  Path B — Bundled sessions (estimated 5-6
    sessions). Stage E + Stage F bundled (D5 + D6
    tagging-only). Stage G + Stage H NOT bundled
    (high architectural content). Stage A standalone
    (Tier 1 Context = 0-baseline new SVG, full
    session warranted). Stage B/C/D standalone (high
    defect counts).

CLAUDE RECOMMENDATION: Path A (one diagram per
session, 8 sessions).

DATA JUSTIFICATION:
  - Master Defect Register entries per stage range
    from 1 (D5 if T-2+ tagging is the only work) to
    7 (D2 Trading Pipeline). Average ~5
    defects/stage.
  - At ~5 defects/stage with full SVG rewrite + 0500000
    prose amend + spec cross-check, a single chat
    session is at moderate-high context-window load
    even with focused scope.
  - Bundling D5+D6 doubles defect count and context-
    window load to closely above the comfortable
    operating envelope. ERROR-TB00149-02 (SVG-DROP-
    IN-BINARY-GENERATION) is the type of class
    that surfaces under context pressure.
  - Loss-minimization: 8 sessions × low risk per
    session > 5 sessions × moderate risk per
    session. Cost of one regression > cost of three
    additional sessions.

ALTERNATIVE PATH B TRADE-OFFS:
  - 3 fewer chat sessions (~37% session-count
    reduction)
  - But: each bundled session carries 2× defect
    count and corresponding regression risk
  - And: bundling encourages silent narrowing (RR-
    SCOPE-007 risk) when context pressure mounts
  - And: a partial bundled-session failure
    requires un-bundling on retry, costing more
    sessions than Path A in the failure case

BILL ACTION REQUIRED:
Ratify Path A (8 sessions) or Path B (bundled).
If Path B: specify exactly which stages bundle.

DOWNSTREAM IMPACT IF DEFERRED:
TB00154 NSI will assume Path A by default per RR-
SCOPE-005 (next logical step from data — 8 stages
in plan = 8 sessions). If Bill chooses Path B at
TB00154 open, TB00154 NSI requires re-issue.
Recommend ratification at TB00153 close.
