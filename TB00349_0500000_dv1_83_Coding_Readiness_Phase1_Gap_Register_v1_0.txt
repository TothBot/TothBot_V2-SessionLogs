3. METHODOLOGY
============================================================

3.1 AUDIT CRITERION — TB00000 §3.2 FIVE QUESTIONS.

Per TB00000 §3.2 (every specification document must
answer five questions): for every component (module,
gate, layer, regime, contract, etc.) defined in the
coding-source corpus:

  Q1. What does this component DO? (inputs -> outputs)
  Q2. What are its interfaces to other components?
  Q3. What are its hard rules? (must / must not)
  Q4. What triggers it and what does it trigger?
  Q5. What does it log?

If any of these five questions cannot be answered from
the document corpus, the spec is INCOMPLETE for coding
per TB00000 §3.1.

3.2 METHODOLOGY GAP IN PRIOR SUB-CAMPAIGN AUDITS.

The DEC-012(a-extended) sub-campaign audits (AA=04 +
AA=06 + AA=07 + AA=08) used a FIVE-BUCKET PARITY-AND-
NAMESPACE methodology:
  bucket (a) 0500000 cross-document parity gaps
  bucket (b) §9 mirror gaps
  bucket (c) 0311001 namespace gaps
  bucket (d) PRESERVE-with-housekeeping disposition
  bucket (e) source-spec observations

NONE of those audits applied the §3.2 five-questions
criterion. The "FULL CLOSURE" status currently
declared for all four AA-domain sub-campaigns reflects
PARITY COMPLETION, not CODING-READINESS COMPLETION.

This is the META-GAP. Phase 2 code rebuild from the
current corpus could fail despite all sub-campaigns
declared CLOSED because the closure criterion was the
wrong criterion for the actual mission.

3.3 AUDIT UNIT.

The audit unit is the COMPONENT CLASS instance — every
mod:* / gate:* / layer:* / regime:* / contract:* /
channel:* / evt:* / rule:* identifier that names a
runtime artifact Claude must code. For each component
class instance, the audit checks Q1-Q5 answerability
across the aggregate coding-source corpus.

3.4 SAMPLE-BASED ASSESSMENT.

Full per-component-instance audit against Q1-Q5 across
all ~150 component instances in 0500000 dv1_83 exceeds
single-session tool budget per RR-SCOPE-011. Phase 1
audit applies the criterion to a REPRESENTATIVE SAMPLE
covering the highest-impact component classes (modules,
gates, layers) and extrapolates findings + identifies
systematic gap patterns. Phase 2 (full per-component
audit) is a carry-forward per Phase 1 disposition.
