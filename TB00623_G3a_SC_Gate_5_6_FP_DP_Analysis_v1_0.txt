============================================================
2. WHY THE FENCE EXISTS (sec 4.1 -- establish purpose BEFORE removal)
============================================================
SC-Gate-5/6 are hard-number-era artifacts. In the pre-CIATS design,
exposure was bounded by FIXED per-trade sizing (e.g. the 0211001 "5% fixed
per_trade_pct" seed) plus FIXED count caps (max_concurrent; an open-order
cap). With a fixed slice size, capping the COUNT of open positions/orders
was the mechanism that bounded total simultaneous exposure and bounded the
order-rate footprint against the exchange. That was their loss-prevention
purpose: a proxy ceiling on aggregate exposure and on order-rate pressure.

The paradigm that gave them meaning is gone. Sizing is CIATS-owned off the
module wallet balance ($5,000 paper seed), starting $0.50/trade, with the
SACRED net 1:1.5 R:R floor the only hardcoded value and the wallet balance
the sole sizing boundary (TB00000 v2_84 sec 7; DEC-115). max_concurrent
was already retired on exactly this reasoning (1.2).

============================================================
3. MINIMIZE-LOSS TEST -- what unique loss does a COUNT gate prevent?
============================================================
For each loss vector a count cap nominally guarded, identify the structural
boundary that already owns it under CIATS-ownership:

  Over-exposure beyond the wallet
    -> PREVENTED by the wallet boundary itself. concurrent count =
       wallet_balance / per_trade_size (1.2). Once the wallet is deployed
       the (N+1)th position cannot be funded. The cap is emergent and self-
       limiting; a count gate is a redundant proxy for "do not exceed the
       wallet," now enforced by arithmetic.
  Single-position / correlated concentration
    -> OWNED by Gate-7 concentration_limit_per_module + exposure_limit_pct
       (DEC-115), expressed in WALLET FRACTION -- the economically correct
       unit. A raw count is an inferior proxy: 20 positions x $0.50 is
       trivial exposure; 2 x $2,500 is the whole wallet. Count does not
       track risk; wallet-fraction does.
  Runaway drawdown
    -> PREVENTED by session-pause 5% / full-halt 10% (per-wallet).
  Sub-floor trades
    -> PREVENTED by rule A1 + Sacred 1:1.5 floor.
  Exceeding Kraken order/rate limits
    -> PREVENTED by the C-2 budget + 80-orders/pair Kraken ceiling +
       rl_critical 0.95 entry-suppression (1.4) -- at the correct layer.
  asyncio dispatch race
    -> PREVENTED by the Risk_Engine BoundedSemaphore (1.3), independent.

RESULT: there is NO residual loss that a Selection-Controller COUNT gate
uniquely prevents. Every vector is already covered by a structural boundary
or by the layer that owns it more correctly.

============================================================
4. OPTION SPACE (DD-011, full set evaluated at turn 1)
============================================================
OPT-1 RETIRE both gates as redundant.
OPT-2 KEEP both as CIATS-owned NON-BINDING seeds (e.g. 10,000) per the
      DEC-115 precedent.
OPT-3 KEEP one, retire the other.
OPT-4 RE-EXPRESS as a different mechanism.

OPT-2 rejected. The DEC-115 precedent does not transfer. Concentration/
  exposure seeds are in WALLET FRACTION -- a real, ownable risk knob CIATS
  can later bind (e.g. 40% of wallet) if data shows concentration causes
  loss. A position-COUNT or order-COUNT seed is NOT a meaningful CIATS knob:
  the wallet/size arithmetic already fixes the count, so a count seed can
  never bind without contradicting wallet-is-sole-boundary, and any value
  set is noise that exists only to always pass. It adds two CIATS-owned
  parameters with zero decision content, re-introduces the count-ceiling
  concept v2_29 deliberately retired, and -- for open orders -- creates a
  SECOND, non-authoritative copy of Kraken's 80/pair limit that can DRIFT
  from the exchange truth (a new loss surface). Fails the completeness
  test's spirit (no meaningless values).

OPT-4 rejected. Every legitimate function is ALREADY correctly expressed
  elsewhere (sec 3): wallet boundary, Gate-7 fraction limits, BoundedSema-
  phore, Kraken C-2 budget. Re-expression would only relocate redundancy.

OPT-3 rejected. No asymmetry justifies keeping one. Both are count proxies;
  both are fully covered. If anything open-order count is the MORE clearly
  redundant (hard Kraken 80/pair + rate budget), and concurrent-position
  count's ceiling was the one explicitly retired in v2_29. Neither has a
  residual unique function.

OPT-1 is the first-principles + minimize-loss-correct path. Retiring both
  removes no loss protection (all protections live in correct authoritative
  layers, sec 3) and is strictly LOWER loss-risk than OPT-2: it eliminates
  the open-order-count drift surface and removes two "always-pass" gates
  that are themselves a completeness defect (a gate that gates nothing).
  No edge case increases exposure: the (N+1)th position is unfundable by
  arithmetic; the 81st order is rejected at the exchange/rate layer; the
  dispatch race stays guarded by the semaphore.

rl_critical 0.95, and the Risk_Engine BoundedSemaphore.

This is a single compliant path (DD-011 OUTCOME 1): both surviving options
are loss-equivalent on protection, and RETIRE is strictly better on drift-

5.1 DATA-COLLECTION CONSIDERATION (Bill-ratified)
============================================================
Question raised (Bill): separate from the controller role, are SC-Gate-5/6
counts worth COLLECTING as CIATS analysis data? Verified in HEAD:
- Concurrent-position count already exists as live Position_Mirror state
  param positions_held_count_per_module (consumed by Gate-7 CHECK_2
  concentration). It is NOT created by SC-Gate-5 and is NOT lost by
  retiring the gate. Moreover the CIATS per-close record (23-field schema,
  D5/D6 canonical) carries entry_timestamp_utc + exit_timestamp_utc per
  trade, so concurrent-position-count-at-entry is EXACTLY reconstructable
  post-hoc from the durable trades_*.jsonl corpus (same-module records
  whose [entry,exit] interval spans a given entry). No dedicated field is
  needed; crowding analysis is already enabled at zero schema cost.
- Open-order count is NOT a tracked state token and is NOT in the 23-field
  record, and is NOT reconstructable from the trade-close corpus (unfilled/
  expired orders never become close records). Capturing it would require a
  new tracked state + a new record field; its independent signal is weak
  (largely collinear with positions-held + in-flight entries).
DECISION (BILL): retire both gates; do NOT add any count instrumentation.
Small analytical upside does not justify the schema/state bloat;
concurrent-count crowding analysis remains available via reconstruction.
This is orthogonal to and does NOT change the retire recommendation. The
next D2 slice is a PURE REMOVAL for G3(a) -- no observation field added.
