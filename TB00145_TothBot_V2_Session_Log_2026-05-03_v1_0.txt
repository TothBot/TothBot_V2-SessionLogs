3. DESIGN PRINCIPLE CODIFIED
============================================================

TothBot V2 paper mode and live mode share ONE execution
code path. The minimum divergence set between modes is
restricted to four specific points:

  (1) Order-dispatch boundary (six methods in WSManager).
      Paper simulates locally via _simulate_entry_fill;
      live transmits via _send_private. Methods:
      add_order, batch_add, cancel_order, amend_order,
      batch_cancel, dispatch_market_sell.

  (2) Step 11 capital-ledger initialization source.
      Live initializes spot_usd_balance and
      portfolio_baseline_USD from snap_orders
      spot_usd_balance. Paper initializes both from
      paper_starting_balance ($1,000 per D-05).

  (3) Step 11 + reconnect snap_orders Mirror handling.
      Live reconciles Position Mirror against
      snap_orders. Paper logs snap_orders for parity-
      of-data-path verification but does NOT populate
      paper Mirror (paper Mirror is owned by the local
      simulated-fill path).

  (4) Capital-ledger update source.
      Live updates spot_usd_balance from balances
      events. Paper updates from simulated-fill P&L
      applied locally per fee schedule.

ALL OTHER SUBSYSTEMS BYTE-IDENTICAL between modes:

  Public WS shards (connect, subscribe, receive).
  Private WS connection.
  WS auth token acquisition (TRADE key, both modes).
  All subscriptions: instrument, status, ohlc, ticker,
    executions, balances.
  Receive loops, JSON ping (every connection),
    zombie monitor (every connection), sequence-gap
    detection (executions and balances), ratecounter
    monitoring.
  snap_orders / snap_trades reception (both modes;
    paper logs, live reconciles).
  Indicator computation, pipeline gates (all 8),
    regime engine, exit triggers (Layers 1a/1b/2/3),
    alerts, drawdown computation logic, reconnect
    (per-shard and private), CIATS data flows.

TRADE key role in paper mode: TRADE key is REQUIRED
in both modes for private WS authentication
(GetWebSocketsToken). TRADE key has both order-
dispatch capability and read capability. In paper
mode, the order-dispatch capability is gated at the
§13.2 dispatch boundary (six methods); the read
capability is exercised in paper for parity-of-
data-path verification.

DATA key role: public WS subscriptions and REST
market data calls (GetOHLCData, REST Ticker).
Used in both modes.

============================================================

The defect is a multi-session governance failure
inherited across at least seven spec versions:

  0511001 dv1_1 (introduced HR-WM-022 narrow parity)
  0511001 dv1_2 (HR-WM-022 preserved; HR-WM-026 narrow
                 parity codified)
  0511001 dv1_3 (HR-WM-022 + HR-WM-026 preserved)
  0511001 dv1_4 (HR-WM-022 + HR-WM-026 preserved)

  1011002 dv1_15 onward (inherited HR-WM-022 / HR-WM-026
                         language; WM-RECONNECT-019
                         added in dv1_21 codifying

                         side reconnect steps)

  0211005 dv1_1 (PT-MODE-003 had CORRECT plain-language
                 reading but contradicted HR-WM-022;
                 contradiction was never escalated;
                 PT-MODE-004 introduced its own
                 contradiction with 0511001 baseline-
                 source semantics; PT-MODE-005
                 codified the narrow Steps-2-8-only
                 parity)

  1011014 dv1_7 (Step 8 "Private WS absence does not
                 affect REST warm-up" reinforced the
                 incorrect paper-mode-no-private-WS
                 reading)

The narrow-parity reading conflated "no order
dispatch" with "no private WS connection." These
are different things. Paper trading exists to
validate the entire system before real capital is
at risk. Validation of public WS only is partial
validation; the private WS data path (token, connect,
executions/balances subscribe, snap_orders, sequence-
gap detection, ratecounter, private ping, private
zombie, private reconnect) is precisely what paper
trading must validate. Skipping it defeats the
purpose.

Bill's plain-language EXACTLY instruction
(consistent across multiple session-open statements)
was honored in spec narrative but violated by the
hard-rule encoding. Multi-session governance
failure.

The defect surfaced when Claude noticed during
STREAM 5 deploy preparation that the deployed code
behavior contradicted Bill's standing emphasis
"paper trading must mimic live trading EXACTLY."
The contradiction was raised before any deploy
command issued.

  0511001 dv1_5: ADR §18 codifies single execution

                 retires HR-WM-022; rewrites HR-WM-026;
                 adds HR-WM-033/034/035.
  1011002 dv1_26: mirrors ADR §18 in coding contract;
                  retires WM-RECONNECT-019; adds
