5.1 ERROR-TB00130-01 — .docx validation defect
------------------------------------------------------------

Severity:   MED (delivery defect caught by Bill).
Rule cited: Not a RR-SCOPE / RR-RULE / RR-SHELL
            violation. Quality/authoring defect.

Defect description:
  Initial .docx build via python-docx 1.2.0 produced a
  file with word/settings.xml missing the required
  w:percent attribute on the <w:zoom> element. OOXML
  schema requires w:percent on <w:zoom>; python-docx
  omits it by default. This defect is invisible on
  file size or structure inspection but causes Word
  to reject the file as "cannot be opened" and causes
  browser download pipes to abort or mangle the
  transfer.

Root cause:
  python-docx library bug: writes <w:zoom w:val="bestFit"/>
  instead of the schema-required <w:zoom w:val="none"
  w:percent="100"/>. Known upstream issue, not a Claude
  authoring error per se, but Claude's delivery workflow
  did not validate before first delivery.

Recurrence context:
  Bill's response: "We have been through this before
  numerous times." Prior chat history (search:
  "0500000 docx won't open download file format issue")
  returned a 03-25-2026 (TB00006-era) resolution that
  said ".docx fails to download cleanly; deliver .pdf
  instead." That prior fix did not identify the
  validation defect as root cause — it worked around
  by falling back to .pdf. TB00130 root-caused the
  underlying issue.

Corrective taken this session:
  (a) Ran /mnt/skills/public/docx/scripts/office/
      validate.py on the initial build. Output:
        "word/settings.xml: 1 new error(s)
         - Element 'w:zoom': The attribute 'w:percent'
           is required but missing."
  (b) Unpacked the .docx via unpack.py, edited
      word/settings.xml to set
        <w:zoom w:val="none" w:percent="100"/>
  (c) Repacked via pack.py. Second validation: all
      PASSED.
  (d) Round-trip tested via LibreOffice PDF conversion:
      opens cleanly.
  (e) Fixed .docx replaced the broken one in outputs.

Standing-rule implication (proposed):
  Add to Claude authoring workflow: every .docx produced
  by python-docx MUST pass validate.py BEFORE delivery.
  This is a quality gate, not a format requirement.
  Deferred to TB00131+ as a governance decision (NOT a
  standing rule authored this session — per RR-SCOPE-001
  Claude does not silently expand governance).
