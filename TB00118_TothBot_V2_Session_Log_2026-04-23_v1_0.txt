2.3 OI-028 root cause — REFINED and LOCKED
------------------------------------------------------------

TB00117 §2.5 Candidate (A) was CORRECT at the architectural
level (reconnect path does not honor paper-mode private-WS
skip) but MIS-IDENTIFIED the failure locus.

TB00117 hypothesis: failure at Step 2b (private WS connect).
Reality:           failure at Step 1 (REST auth token).

The Kraken API key installed on the VPS is currently being
rejected by Kraken with `EAPI:Invalid key`. In paper mode
the key is not exercised anywhere else (order dispatch
suppressed, balance static from config, executions channel
unused) — so the rejection is operationally harmless AS
LONG AS the code does not attempt to use the key.

`_execute_reconnect_sequence` unconditionally calls
`_rest_get_ws_token()` at Step 1 before any branching. That
call signs the request with the rejected private key and
Kraken returns EAPI:Invalid key. The call raises; the
reconnect-loop catches the exception as
RECONNECT_ATTEMPT_FAILED, consumes an attempt, and retries.
Every attempt fails identically. After
MAX_RECONNECT_ATTEMPTS (20) consecutive identical failures,
the loop exhausts and emits FATAL_RECONNECT_FAILURE →
WS_MGR_FATAL → systemd restart.

This is a persistent-failure class, not transient. The
dv1_21 budget extension from N=10 to N=20 did not help
because extending a budget does not change a deterministic
fail-every-attempt outcome.

ROOT CAUSE: `_execute_reconnect_sequence` lacks the paper-
mode early-return branch that `_startup_connect_and_subscribe`
has (FIX-WM-PT-001, dv1_14). The private-WS-skip is
implemented on the startup path but not on the reconnect
path. 0511001 dv1_1 §82-84 and §128-129 already stated the
intended behavior at the specification level; the code was
out of compliance with the spec.
