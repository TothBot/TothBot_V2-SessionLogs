1. THE GENUINE WARM-UP DEFECT - FIXED (Code 2cd43a4, 1158 -> 1164 green, pushed + DR-010)
   ROOT CAUSE: the REST rate control was at the WRONG SCOPE. WarmupOrchestrator.warm_all fans out one
   coroutine PER PAIR (asyncio.gather); the only throttle was a per-pair 1.1s sleep that spaces a SINGLE
   pair's own two calls. So a large universe fired ~one REST call per pair at once (bounded only by the
   aiohttp connector to ~5-10 concurrent = a sustained ~50-100 calls/s drain) - far over Kraken's ~1/s
   public budget -> mass rate-limit -> mass WARM_UP_FAIL -> a large fraction of the universe silently
   never seeds (a systemic false-negative). Invisible at 3 pairs (6 calls), dangerous at 735.
   THE FIX: new mod rest/rate_limiter.py RestRateLimiter - a single shared leaky-bucket pacer (rate
   1/min_interval, no burst). Every _public/_private call in KrakenRestClient now awaits this ONE
   governor, so calls are spaced >= min_interval GLOBALLY no matter how many coroutines call concurrently
   (warm-up + regime + liquidity + the periodic reconcile all honor the one per-IP budget). asyncio.Lock
   serializes the slot + is held across the spacing sleep (N acquirers released one per interval). AIMD
   back-off: _maybe_penalize pushes the next slot out on a Kraken rate-limit/too-many response. Default
   1.1s (AR-036, an engineering cadence NOT a seed); min_interval 0 = no-op (test-fast). PURE save the
   injected clock+sleep. Minimal blast radius - the existing per-phase staggers are left (harmless double-
   pacing); no trading-latency impact (dispatch is WS not REST). Tests: test_rate_limiter.py (zero-interval
   no-op, spacing, no-burst-credit, penalize, N concurrent released one/interval) + test_rest.py injects a
   zero-interval limiter. NO NEW SEED.
