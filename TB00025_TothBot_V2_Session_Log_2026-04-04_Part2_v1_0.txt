2.2 Lifecycle Model Correction — 0311003 Was Wrong
------------------------------------------------------------

Bill identified that 0311003 VP-004 was incorrect.
Correct lifecycle model confirmed by Bill:

  dv: Working draft. Under constant review. Bill may
      approve directly dv → v. No rv step required
      for initial document development.

  dv → v: Bill approves directly. Version resets to v1_0.

  v → rv: Required path when an approved document needs
          changes. BILL ACTION ONLY.

  rv → v: Bill re-approves after revision. v1_0 reset.

  v → v:  Minor corrections only. Bill approval.

  v → rt: Retirement. BILL ACTION ONLY.

The rv state is NOT used for initial document development.
It is ONLY used when revising an already-approved document.

This correction was applied to:
  0311003 VP-004 and VP-005 (full rewrite)

------------------------------------------------------------
2.3 git add . Is Too Broad — Explicit add Required
------------------------------------------------------------

The Gate 1 commit script used `git add .` which picked
up all files already in the local repo folders — not
just the 10 Phase 1 documents. This caused premature
GitHub commit of WD coding specs and all 0500000
previous versions.

STANDING RULE: All future commit scripts MUST use
explicit `git add [filename]` for each intended file.
Never use `git add .` in TothBot V2 commit scripts.

------------------------------------------------------------
2.4 Multi-Line Commit Messages Break in Windows Batch
------------------------------------------------------------

The multi-line git commit -m message in Gate1_Commit.bat
caused the commit message body to be interpreted as
commands, creating a spurious file named "v" in the repo.

Fix applied: `git revert HEAD --no-edit` removed the
spurious file. All documents remain correctly committed.

STANDING RULE: Windows batch git commit messages must
use single-line -m strings or a separate message file.
Multi-line commit messages do not work in .bat files.
