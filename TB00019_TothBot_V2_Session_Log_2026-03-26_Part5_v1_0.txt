NEW-01: amended exec_type MISSING from dispatch table
------------------------------------------------------------

SOURCE: Kraken WS v2 executions API documentation (confirmed
from Search 1 and Search 2).

FINDING: The complete exec_type enum in the Kraken
executions channel is:
  pending_new, new, trade, filled, iceberg_refill,
  canceled, expired, amended, restated, status
That is 10 exec_types total.

WM-DISP-023 (1011002 dv1_9) lists only 9:
  pending_new, new, trade, filled, iceberg_refill,
  canceled, expired, restated, status
  MISSING: amended

AR-023 (0500000) specifies "all 10 exec_type values handled
explicitly." This requirement was not met because amended
was not in the dispatch table.

WHAT amended MEANS (from API docs):
"amended: There is a user-initiated amend on the order,
i.e. limit price change."

WHEN TothBot ENCOUNTERS amended:
For TothBot's positions, amended events occur when:
(a) amend_order response arrives for an amend TothBot
    itself sent (e.g., amending TP price via Layer 1a
    trailing logic — future feature). For V1 with static
    TP, TothBot does not amend its own orders voluntarily.
(b) Kraken engine amends an order internally (rare —
    this is restated, not amended).
(c) If an operator manually amends an order via the
    Kraken web UI (should not happen in production).

RESOLUTION: Add amended to WM-DISP-023 dispatch table:
  exec_type == "amended" → log UNEXPECTED_ORDER_AMENDED,
    alert operator HIGH (TothBot V1 does not amend orders
    voluntarily — any amended event is unexpected and
    requires investigation), check if the amended order
    is a TP or emergSL and log the new price.
Assign: AR-057.

FP/DP ANALYSIS: TothBot does not use amend_order in V1
(amend_order is in the spec for future use). An amended
event arriving when TothBot did not initiate an amend
indicates external interference (operator via web UI)
or a future bug. Logging and alerting is loss-protective.

LOSS MINIMIZATION RATIONALE: Silently ignoring an amended
event could mean an TP or emergSL price was changed
without TothBot's awareness, breaking the sacred R:R.
Alert is mandatory.

NEW-02: websockets library max_queue not specified
------------------------------------------------------------

SOURCE: websockets v16.0 API reference (Search 8, ref 73).

FINDING: The websockets library has a `max_queue=16`
default parameter. This is the internal receive buffer
between the TCP stack and the application's `recv()` calls.
With 16, if TothBot cannot process 16 queued messages fast
enough (unlikely but possible during burst events —
startup instrument snapshot for 200+ pairs, rapid ticker
events on multiple open positions), messages could be
dropped silently or the connection could back-pressure.

For TothBot monitoring 50+ pairs with multiple subscriptions
(ticker, ohlc, executions, balances, instrument, status),
burst message counts can exceed 16 during:
  - Initial instrument snapshot (200+ pairs)
  - Rapid market moves (many bbo ticker events)
  - Post-reconnect snapshot processing

RESOLUTION: Set `max_queue=None` (unlimited) on all
websockets connections. The asyncio event loop processes
messages as fast as possible. max_queue=None removes the
artificial ceiling. Memory usage is minimal (each message
is a small JSON payload).
Assign: AR-058.

FP/DP ANALYSIS: The 5+ sources on websockets library
configuration confirmed that max_queue controls the receive
buffer. Setting to None eliminates any possibility of
message drops due to buffer overflow. The alternative
(leaving default 16) creates a silent failure mode under
burst conditions — unacceptable for a 24/7 trading system.

LOSS MINIMIZATION RATIONALE: A dropped executions channel
message = missed fill = Position Mirror desync = loss risk.
max_queue=None is mandatory.

NEW-03: Library ping vs application JSON ping distinction
------------------------------------------------------------

SOURCE: websockets v16.0 API reference (Search 8, refs
73, 74, 76). WM-WSLB-001 and WM-WSLB-002 (1011002 dv1_9).

FINDING: The existing spec mixes two distinct ping
mechanisms without clearly distinguishing them:

  MECHANISM 1 — Library TCP-level ping (WebSocket protocol):
    websockets.asyncio.client.connect() has `ping_interval`
    and `ping_timeout` parameters. When set, the library
    sends WebSocket PING frames automatically. This is
    transparent to TothBot's application code.
    WM-WSLB-002 sets `ping_interval=30`.

  MECHANISM 2 — Application-level JSON ping (Kraken WS v2):
    A-7 specifies that WS Manager sends {"method":"ping"}
    every 30 seconds and expects {"method":"pong"}.
    This is TothBot's own code, tracked at application level.

PROBLEM: Current spec (WM-WSLB-002) says "websockets
library ping_interval=30" but does not clarify the
distinction. A reader would not know whether TothBot relies
on the library's transparent ping or the JSON ping, or both.

The TWO serve different purposes:
  Library ping: maintains TCP connection, transparent,
    no application involvement
  JSON ping: application health check. TothBot tracks pong
    response. No pong within 10s = dead connection
    (zombie detection complement to last_real_data_time).

RESOLUTION: Specify both mechanisms explicitly:
  1. Library: ping_interval=None (DISABLE library's built-in
     ping). Rationale: TothBot's application-level JSON
     ping already covers keepalive. Running two concurrent
     ping mechanisms is redundant and can create confusion.
     TothBot has full visibility into the JSON ping/pong
     exchange; it does NOT have visibility into the library's
     transparent PING frames. Having only the JSON ping gives
     TothBot single, authoritative keepalive visibility.
  2. Application: JSON ping {"method":"ping"} every 30s.
     Pong expected within 10s per A-7. No pong = reconnect.
     This is the sole keepalive mechanism.
Assign: AR-059.

FP/DP ANALYSIS: Disabling library ping simplifies the
connection state model. TothBot tracks application-level
pong which feeds zombie detection. Duplicate pings at the
TCP level add noise without improving detection.

LOSS MINIMIZATION RATIONALE: Single authoritative keepalive
mechanism with full application visibility reduces the
chance of a half-alive connection being missed.

NEW-04: websockets API changed — new asyncio import path
------------------------------------------------------------

SOURCE: websockets v14+ documentation (Search 8, refs
73, 75, 76). The library switched to a new asyncio
implementation in v14.

FINDING: In websockets v14+, the new stable API is:
  from websockets.asyncio.client import connect
  async with connect(uri, ...) as websocket: ...

The old API:
  import websockets
  async with websockets.connect(uri, ...) as websocket: ...
  is being deprecated (still works in v14-v16 but will
  be removed in a future version).

TothBot's spec references `websockets.connect()` in several
places (SS-STARTUP-006, SS-STARTUP-013, WM-RECONNECT-006)
but does not specify the import path. This will break when
the legacy API is removed.

RESOLUTION: All websockets connection code MUST use:
  from websockets.asyncio.client import connect
  conn = await connect(uri, ...)
Not the legacy:
  import websockets
  conn = await websockets.connect(uri, ...)
This applies to all four WS connections:
  public WS (wss://ws.kraken.com/v2)
  private WS (wss://ws-auth.kraken.com/v2)
  and their reconnect variants.
Assign: AR-060.

FP/DP ANALYSIS: Forward compatibility is a reliability
requirement for 24/7 operation. A library update that
removes the legacy API would silently break TothBot if
the code still uses the old import path.

LOSS MINIMIZATION RATIONALE: Unplanned system outage due
to library API change during live trading = loss event.

NEW-05: Kelly formula must use NET R:R from Trade Outcome Bus
------------------------------------------------------------

SOURCE: Multiple Kelly criterion sources (Search 10,
refs 92-101). CIATS-KE-001 (1011010 dv1_2).

FINDING: The Kelly formula `K_full = W - ((1-W)/R)` where
R = avg_reward_risk_ratio. This is not currently specified
as NET or GROSS. Ambiguity risk: if CIATS uses the
gross_target/mae_pct ratio (before fees) as R, the Kelly
size would be inflated.

CORRECT SPECIFICATION: R must be the REALIZED NET R:R
from the Trade Outcome Bus. The Trade Outcome Bus records
actual P/L including all fees. Therefore:
  avg_net_rr = mean(net_gain_usd / net_loss_usd)
               over all closed trades in rolling window
  where net_gain_usd and net_loss_usd already include
  all entry maker fees, exit taker fees, and are the
  actual USD captured/lost per trade.
Not the theoretical gross targets.

Additionally: W (win rate) = trades where net_gain_usd > 0
divided by total closed trades. Not gross TP fills — only
trades that were net positive after fees.

RESOLUTION: Clarify CIATS-KE-001 that both W and R are
computed from the Trade Outcome Bus net P/L data, not from
the R:R formula targets.
Assign: AR-065.

FP/DP ANALYSIS: For TothBot V1, trades target 1:1.5 net
R:R, so realized R should cluster near 1.5 for winning
trades and near -1.0 for losing trades (MAE exits). Using
net P/L from Trade Outcome Bus is the correct and only
valid source for Kelly inputs.

LOSS MINIMIZATION RATIONALE: Inflated Kelly from gross
R:R inputs = oversized positions = excess loss risk on
losing trades. Net R:R is the only valid input.

NEW-06: Logger implementation — queue.Queue not asyncio.Queue
------------------------------------------------------------

SOURCE: Python asyncio logging best practices (Search 5,
refs 43-51). Python logging cookbook (ref 48).

FINDING: The existing spec (AR-014, 0500000 Section 6)
says "Logger async bounded queue, 10,000 entries, separate
async writer flushes to disk." This is ambiguous and could
lead to an incorrect implementation using asyncio.Queue
with an asyncio coroutine writer.

THE CORRECT PATTERN (Python standard library):
  1. Bounded queue: queue.Queue(maxsize=10000)
     NOT asyncio.Queue (not thread-safe for QueueListener)
  2. QueueHandler: logging.handlers.QueueHandler(std_queue)
     Attached to the Logger instance on the hot path.
     put_nowait() behavior — raises queue.Full if full.
  3. QueueListener: logging.handlers.QueueListener(
         std_queue, FileHandler(...))
     Runs in a SEPARATE THREAD (not asyncio coroutine).
     Automatically handles reading from queue and writing
     to file in its own thread context.

WHY NOT asyncio.Queue + asyncio writer coroutine:
  - asyncio.Queue is not safe for use with QueueListener
    which runs in a separate thread
  - An asyncio writer coroutine competes with trading tasks
    for the event loop — cannot guarantee timely execution
  - The QueueListener pattern was designed exactly for this
    use case: non-blocking log production + threaded consumer

WHY queue.Full on put_nowait() is the right choice:
  If queue fills (10,000 messages), it means the disk
  writer is backed up. This is a system health signal.
  TothBot MUST: catch queue.Full, log to stderr, alert
  operator, and continue without blocking the hot path.

RESOLUTION: Specify Logger implementation explicitly:
  - queue.Queue(maxsize=10000) — Python standard library
  - logging.handlers.QueueHandler — on asyncio hot path
  - logging.handlers.QueueListener — in separate thread
  - On queue.Full: log to stderr, alert HIGH, skip the
    log record rather than block
  - Thread must be daemon thread (does not block shutdown)
Assign: AR-063. Assign to new 1011007 Logger_Coding_Spec.

FP/DP ANALYSIS: 6 independent Python sources confirmed
the QueueHandler+QueueListener pattern is the standard
non-blocking logging approach for asyncio applications.
The asyncio.Queue + asyncio writer coroutine would be
incorrect because (a) QueueListener requires a queue.Queue
compatible interface from the standard library and
(b) coroutine writers create event loop scheduling problems.

LOSS MINIMIZATION RATIONALE: Logger blocks hot path =
delayed pipeline execution = missed exits = loss risk.

NEW-09: Exit Controller Layer 1a — three definable triggers
------------------------------------------------------------

SOURCE: FP/DP analysis from momentum exit trigger research
(Searches 3 and 4), cross-referenced with TothBot
architecture (0500000 dv1_13).

FINDING: OI-003 (Exit Controller full specification) is
pending and blocks Phase 5-6 work. However, three Layer 1a
exit triggers are definable independently of the SSS signal
specification (OI-001) that also remains pending.

These three triggers are derived from first principles:
each is the symmetric reversal of an entry gate condition.

TRIGGER EC-L1A-001: HTF Regime Reversal Exit
  Exit condition: 1H EMA(20) crosses BELOW EMA(50) while
    a long position is open.
  Rationale: Gate 4 (HTF Confirmation) requires 1H
    EMA(20) > EMA(50) for entry. If this condition reverses
    after entry, the hourly trend that justified the entry
    no longer holds. The position should exit cleanly at
    the TP price. If TP is not reached, signal-based exit
    applies: cancel TP, cancel emergSL, market sell.
  Timing: Checked on every ohlc(60) candle close event
    for pairs with open positions.
  Source validation: Multiple sources confirm hourly
    trend reversal (EMA crossover) as a primary momentum
    exit trigger for 5-minute systems (FMZQuant, Collin
    Seow, Highstrike — Search 4 refs 34, 36, 35).

TRIGGER EC-L1A-002: Daily Regime Downgrade Exit
  Exit condition: At 00:00 UTC regime refresh, pair
    reclassified from TRENDING_POS → TRENDING_NEG or
    NON_DIR+ELEVATED while a long position is open.
  Rationale: Gate 3 (Regime Pre-Filter) blocks new entries
    in TRENDING_NEG and NON_DIR+ELEVATED. If the overnight
    regime computation reclassifies an open position's
    pair to a NO-ENTRY regime, the position is now in
    a hostile environment. Exiting reduces exposure.
  Timing: Checked immediately after 00:00 UTC regime
    computation completes, before any new pipeline evals.
  Note: This is a SIGNAL exit, not a MAE exit. The exit
    sequence is: cancel TP, cancel emergSL, market sell.
    No change to sacred R:R (R:R still enforced on the
    partial exit; loss may exceed mae_mult if price has
    moved significantly before regime flipped).

TRIGGER EC-L1A-003: Maximum Hold Time Exit
  Exit condition: Position held for > max_hold_candles
    5-minute candles without TP fill.
  Rationale: Prevents capital lockup in stale positions.
    A position held for many candles without reaching TP
    suggests the entry signal was weak or momentum faded.
  Starting value: max_hold_candles = TBD (OI-003).
    CIATS-owned. Requires FP/DP derivation with SSS signal
    data. Cannot be specified without SSS specification.
    Recommend: leave TBD in EC-L1A-003 until OI-001
    (SSS) is resolved and backtesting provides data.
  Timing: Checked at each 5-min candle close for all
    open positions.

Assign: AR-062. Assign EC-L1A-001, EC-L1A-002, EC-L1A-003
to new 1011005 Exit_Controller_Coding_Spec dv1_0.

FP/DP ANALYSIS: EC-L1A-001 and EC-L1A-002 are derived
from first principles (symmetric reversal of entry gates).
5+ sources confirm EMA crossover reversal as momentum exit
signal (Search 4). EC-L1A-003 requires backtesting data
and is left as TBD pending OI-001.

LOSS MINIMIZATION RATIONALE: EC-L1A-001 and EC-L1A-002
prevent holding positions after the edge has demonstrably
expired. They reduce the holding period in adverse
conditions. EC-L1A-003 releases capital from stale
positions, reducing opportunity cost.

NEW-10: TP partial fill — Position Mirror qty update required
------------------------------------------------------------

SOURCE: Kraken executions API partial fill behavior
(Search 2, refs 11, 12, 15). AR-054 (partial fill on
ENTRY), exec_type=trade documentation.

FINDING: AR-054 handles partial fills on ENTRY orders.
But there is a symmetric case not yet specified: partial
fills on TP EXIT orders.

When a TP limit order partially fills:
  - exec_type=trade arrives with last_qty < TP full qty
  - The position is PARTIALLY closed
  - cum_qty = how much has been closed so far
  - The remaining open qty still needs protection

CURRENT STATE: The spec does not address what TothBot
does when a TP order is partially filled (trade exec_type
for the TP order_id):
  A) Does Position Mirror update to reflect partial close?
  B) Does TothBot amend the emergSL qty to match
     remaining open qty?
  C) Does the Logger record the partial TP fill?

CORRECT BEHAVIOR (derived from first principles):
  On exec_type=trade for TP order:
    1. Calculate remaining_qty = TP_order_qty - cum_qty
    2. Update Position Mirror: position_qty = remaining_qty
    3. emergSL order was placed for full original qty.
       It must be amended to remaining_qty using amend_order
       to avoid selling more than held.
    4. If remaining_qty < qty_min OR remaining_qty * bid
       < cost_min: cancel emergSL, market sell remainder
       immediately (can't protect too-small position).
    5. Log TP_PARTIAL_FILL with original_qty, filled_qty,
       remaining_qty, new emergSL qty.
    6. If exec_type=filled (TP fully filled): cancel emergSL
       as currently specified (L1b). No qty amendment needed.

NOTE: amend_order (+1 rate counter for resting order) is
preferred over cancel+resubmit emergSL (+8 rate counter).

NOTE 2: TP partial fills are uncommon for small position
sizes with adequate liquidity (Gate 2 $500k/day). But
must be handled explicitly — an undetected partial TP
with a full-qty emergSL means an oversized stop relative
to the actual remaining position.

Assign: AR-066. Assign to 0411001, 1011002,
new 1011005 Exit_Controller_Coding_Spec.

FP/DP ANALYSIS: From Kraken API documentation, exec_type=
trade fires on every partial fill for any order type (entry
or exit). The asymmetry between AR-054 (handles entry
partial fills) and the absence of TP partial fill handling
is an architectural gap. Multiple sources confirm partial
fills must be tracked for correct position sizing.

LOSS MINIMIZATION RATIONALE: A full-qty emergSL protecting
a partial position = emergSL fires, sells more than held =
rejected order or accidental short (not possible in spot,
but the rejection itself is a system error event). Must
be correct.
