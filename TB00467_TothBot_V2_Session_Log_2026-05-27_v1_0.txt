2. DECISION: APPROVED CONSOLIDATION ROADMAP (4 PHASES)
------------------------------------------------------------

Root cause of the drift: no master inventory. Each session
picked a consolidation target by feel, re-deriving scope
from memory and NSI carry-forwards. The fix is an audit-
first roadmap with a single living register that every
future session reads, executes the next numbered item from,
and marks closed.

APPROVED ROADMAP:

  Phase 0 — Audit scope definition. One short artifact
            listing every document that could hold trading
            architecture and explicitly excluding the ones
            that cannot (GitHub 10XXXXX, Testing 10XXXXX,
            AA=00/01/02/03/09/11). This is the denominator.

  Phase 1 — Per-document architecture audit. Walk each
            in-scope document; extract every architectural
            element (rule, gate, contract, interface, event,
            channel, architectural-meaning parameter); record
            for each whether it is already in a 0500000
            diagram and which one. Output: the Consolidation
            Register (single source of truth). ONE source
            document audited per session per RR-SCOPE-011.

  Phase 2 — Classification + consolidation plan. Apply
            DD-012 three-bucket test to every element:
            (a) already in a diagram = no action;
            (b) architecture that must move into a diagram;
            (c) non-architectural content that stays
            (parameters, Kraken API detail, test seeds,
            GitHub specs, testing specs). Sequence the
            "must move" items into session-sized batches.

  Phase 3 — Execution. One batch per session, checked off
            against the register, until zero open "must
            move" items remain.

REGISTER-CARRIES-THE-PLAN PRINCIPLE: the plan is NOT a
separate document. The register itself carries a sequence-
number column and a status column. One artifact, not two
(avoids the O(N-squared) drift DD-008 names). Every future
session opens by reading the register and executing the
next numbered open item.

DECOMPOSITION PRINCIPLE: the audit is built one source
document per session, never the whole corpus in one shot
(RR-SCOPE-011). The audit is real extraction work, not
"just reading," and will exhaust a session on the big specs.
