       multiple emitters = FAIL-EN where N>=2).

  2.2  CONSUMPTION COVERAGE: the token appears in at least one component's
       consumes field across all 10 diagrams (zero consumers = FAIL-C0;
       this MAY be acceptable for true external-boundary leaf events such
       as file:tothbot.log writes, but evt:* tokens are internal events
       and zero-consumer is a FAIL by default).

  2.3  PIPELINE-ORDER COHERENCE: where multiple diagrams reference the
       same evt:* token, the producer-diagram must topologically precede
       (or be cross-diagram-linked-to) every consumer-diagram per the
       diagram-tier ordering documented in 0500000 dv1_93 section 1
       (Tier-1 master → D1..D9 subsystem; cross-diagram event linkage
       must be explicit in component descriptors).

  2.4  TOKEN-FORM CONSISTENCY: evt:* token spelling must be byte-
       identical across producer and all consumers (no case drift, no
       suffix variance, no parenthetical mismatch).

Classification scheme (mirrors V-041 Class A/B distinction):

  Class A — token appears in <desc> consumes/produces CSV body but as a
            prose-fragment artifact (whitespace-split token, parenthetical
            remainder, sentence-context residue); REJECT from FAIL scope
            with provenance note.

  Class B — token is canonical evt:* form in clean CSV position; FAIL is
            actionable; emit per-FAIL proposed ADD-only amendment.

FAIL codes:

  FAIL-E0  : evt:* token has zero producers across 10 diagrams
  FAIL-EN  : evt:* token has N>=2 producers across 10 diagrams (emission
             multiplicity violation)
  FAIL-C0  : evt:* token has zero consumers across 10 diagrams (internal
             event with no sink)
  FAIL-PO  : evt:* token producer/consumer pipeline-order violation
             (consumer diagram precedes producer diagram in tier
             ordering with no explicit cross-diagram linkage)
  FAIL-TC  : evt:* token-form inconsistency between producer and consumer
