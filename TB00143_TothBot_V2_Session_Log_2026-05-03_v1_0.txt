3.2 FP/DP -- IS ADDITIONAL PROBE WORK REQUIRED?
------------------------------------------------------------

QUESTION: Probe more before STREAM 4 begins, or proceed
with design-level protection?

FIRST PRINCIPLES:

  FP1  Cloudflare throttles per source IP, NOT per
       WS connection. PATH 2 splits sockets but
       Cloudflare counter sees same VPS IP
       (87.99.141.44) regardless.

  FP2  PATH 2 architecturally fixes per-connection
       limits, NOT per-IP. Clean Phase D run does not
       prove the IP rate limit was respected -- it
       proves whatever the IP limit is, 748 acks in
       ~5-10s of startup phase fits inside it.

  FP3  Per-shard mid-session reconnect risk: low
       (~374 subscribes well-spaced from prior 748).

  FP4  Back-to-back full-restart risk: HIGH and
       unprobed. Bot starts T=0 (748 subs), crashes
       T=2min, restarts T=2min30s (748 more subs).
       1496 subs in 3-min window. Same failure mode
       that killed dv1_22.

  FP5  Probe gives SAMPLE evidence, not BOUND.
       Cannot cover full operational space of restart
       cadences.

  FP6  Per Deming 6.1: correct fix is mathematical.
       dv1_25 self-limits outbound subscribe rate so
       that under any sequence of startup/reconnect/
       restart events, IP-level subscribe count over
       any 10-min sliding window stays below known-
       safe threshold with margin. DESIGN guarantee,
       not sampled observation.

DATA POINTS:

  DP1  Phase D long: 1 startup, 0 restarts, 0
       disconnects observed. No reconnect/restart
       data.
  DP2  Kraken's actual subscribe rate limit not
       documented at WS layer. ~150/10min inferred
       from dv1_22 failure pattern.
  DP3  dv1_25 does not exist yet.
  DP4  DC-PREFLIGHT-006 (NEW TB00142 v1_8) requires
       mathematical analysis BEFORE deploy.
  DP5  systemd StartLimitBurst + StartLimitIntervalSec
       directives are operational config, controllable
       from tothbot.service.
  DP6  Empirical restart probe = multiple hours per
       run for meaningful sample.
  DP7  Loss-prevention: Message Limit Error at startup
       halts trading. Mathematical bound from code
       design defends against ALL restart patterns;
       empirical probe only defends against sampled
       ones.

DECISION: Design path, three layers:

  CODE LAYER: dv1_25 implements outbound subscribe
  pacing token bucket. Conservative target: ~10
  subscribes/sec aggregate.

  OPERATIONAL LAYER: tothbot.service configures
  StartLimitBurst=3 + StartLimitIntervalSec=600
  (max 3 starts per 10 min). Blocks runaway restart
  cycles regardless of bot-internal state.

  GOVERNANCE LAYER: DC-PREFLIGHT-006 Q1-Q5
  mathematical proof BEFORE VPS deploy. Documented
  in dv1_25 Claude Code session log.

STREAM 5 deploy + observation provides empirical
confirmation that the mathematical bound holds in
practice -- using the actual production code, not
probe code.

NO ADDITIONAL PROBE WORK BEFORE STREAM 4.
