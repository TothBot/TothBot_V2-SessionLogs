============================================================
1. FIRST PRINCIPLES -- WHAT THE PARAMETER PHYSICALLY DOES
============================================================

It is the sharding divisor: how many pair subscriptions ride
one public WS connection. Changing it moves exactly five
physical quantities:

  (1) N_conns (connection count for the union universe).
  (2) Per-connection message throughput and processing load.
  (3) Blast radius of a single connection drop (how many
      pairs go blind when one shard dies).
  (4) Proximity to Kraken's real per-connection subscription
      capacity. Overflow failure mode: subscriptions
      silently rejected -> blind pairs -> Signal_Pipeline
      and Exit Controller starved of candle data for those
      pairs -> missed Layer 1a / Layer 2 exits.
  (5) Proximity to the AR-080 Cloudflare reconnect ceiling
      (150 connection-establishment requests per rolling
      10-minute window per source IP). More shards = more
      establishment requests in a reconnect storm. Breach
      locks out reconnection up to 10 minutes, leaving
      Layer 3 emergSL as the ONLY position protection
      (AR-001 + AR-055 invariant chain).

It does NOT touch entry selection, sizing, exits, or any
per-trade economic quantity. It is exchange-interface
plumbing.

============================================================
2. DATA-DRIVEN TEST -- CAN CIATS EVER TUNE IT VALIDLY?
============================================================

CIATS's inference machinery operates on TRADE OUTCOMES:
proposal engine activates at the 200-trade hard floor;
methods are Mann-Whitney U, Sharpe, Spearman, EWMA, CUSUM,
Half-Kelly -- all keyed to per-trade statistics.

SYMBOLS_PER_CONN_SAFE has no per-trade attribution. No
sample of 200 (or 2,000) trade outcomes carries information
about the optimal shard size. The relevant evidence is
infrastructure telemetry (message latency, queue depth,
reconnect counts, subscription-reject events) -- a data
stream CIATS does not model and has no statistical
apparatus for.

CONCLUSION: "CIATS-owned" is unactionable. CIATS could
never assemble valid evidence to propose a change. The
label grants a knob that can only ever be turned blind.

============================================================
3. LOSS-MIN TEST -- WHAT DOES THE KNOB RISK?
============================================================

Both tuning directions are pure tail risk with no profit
pathway:

  RAISE it -> fewer shards, but closer to Kraken's real
  per-connection capacity. The ceiling is not published via
  API; it is discovered by breaching it. Breach = silently
  blind pairs = missed exits = direct loss.

  LOWER it -> more shards -> more establishment requests
  during reconnect storms -> AR-080 breach risk -> up to
  10-minute reconnect lockout with Layer 3 emergSL as sole
  protection.

Expected-return upside from tuning: none identified. The
asymmetry (zero upside, two distinct exit-protection failure
modes as downside) fails the loss-min priority decisively.
This is precisely the class of fence section 4.1 says not to
hand to an optimizer.
