ITEM 4 - REST CANCEL/MODIFY + MASS-CANCEL + KILL-SWITCH + CANCEL-ON-DISCONNECT
================================================================================
ANSWER (Confidence HIGH on the BTP/COD facts):
  (a) CANCEL/MODIFY: BTP (the native binary order-entry API) has only Open(O),
      Attributed-Open(T), Modify(M) send types - NO standalone Cancel. Cancel = a
      Modify with quantity <= filled quantity (qty 0 cancels). FIX modifies in place
      (Cancel/Replace -> ExecType=5 Replaced, same ClOrdId/OrigClOrdId). REST
      documents POST /orders (submit) + GET /orders (query) only; the explicit REST
      cancel verb is NOT surfaced publicly.
  (b) MASS-CANCEL: none documented in BTP/FIX/REST.
  (c) KILL-SWITCH: yes, as a risk-manager control - Position Limits Per Account; "when
      limits are set to zero, only liquidating orders are permitted." Setting the limit
      to zero is the de-facto account kill/flatten lever (blocks new exposure, allows
      only liquidating orders). It is an admin/risk-manager action, not a trader API call.
  (d) CANCEL-ON-DISCONNECT: YES, and it is the FORCED DEFAULT. "If a connection is
      dropped, the respective open orders are canceled" (direct + CQG). BTP Logout
      persistOrders field = "Cancel-on-disconnect is default (unused, reserved)."
      Heartbeat interval in BTP Login (uint8; <=5s defaults to 30s); FIX uses standard
      Heartbeat/Test-Request + HeartBtInt(108).
  CRITICAL CAVEAT for 0500000 13.2/13.3: cancel-on-disconnect cancels OPEN/WORKING
  ORDERS - it does NOT flatten an existing open POSITION. A crashed TothBot drops its
  resting exit orders (good) but a NAKED open position is not auto-liquidated by COD.
  True auto-flatten would rely on the position-limit-to-zero kill-switch (a
  risk-manager control), and whether THAT auto-liquidates a held position vs only
  blocking new orders is unconfirmed.
EVIDENCE:
  1.  bitnomial.com/exchange/docs/api/btp/order-entry/ (Open/Attributed-Open/Modify; cancel via Modify-to-<=fill; no Cancel/Cancel-All)
  2.  bitnomial.com/exchange/docs/api/btp/login/ (heartbeat field; Logout persistOrders = COD default reserved)
  3.  bitnomial.com/exchange/docs/market-operations/order-entry-risk-management/ (COD; position-limit-zero kill-switch; throttle; price bands)
  4.  bitnomial.com/exchange/docs/api/rest/orders/ (POST /orders, GET /orders; no public DELETE/PATCH/mass-cancel)
  5.  bitnomial.com/exchange/docs/api/rest/overview/ (REST v1; authenticated orders endpoint)
  6.  bitnomial.com/exchange/docs/api/fix/overview/ (orders modified in place; Cancel/Replace -> ExecType 5)
