------------------------------------------------------------
2.1 Root Cause — Gate 1 Script Used git add .
------------------------------------------------------------

The Gate 1 commit script (Gate1_Commit.bat, TB00025)
used `git add .` instead of explicit per-file git add.
This caused ALL files already in the local repo folders
to be committed to GitHub, including:

  - 0411001 dv1_10 (WD, premature — no parent chain)
  - All other WD coding specs
  - Multiple 0500000 prior versions (docx, pdf, svg)

Decision at the time (TB00025 Section 2.5): leave as-is.
Files clearly marked dv. No functional harm. No action.

This decision was wrong. It created a technical debt that
surfaced immediately in the Phase 2 AA=04 commit.

------------------------------------------------------------
2.2 Phase 2 AA=04 Commit — What Went Wrong
------------------------------------------------------------

When Phase 2 AA=04 documents were placed in the local
repo folder and git add was run for each file, git
reported nothing to stage. Reason: the 5 AA=04 files
(0400000, 0410000, 0411001 dv1_11, 0411002, 0411003)
were already on GitHub from the premature Gate 1 commit.

Git considered them already tracked and up to date.
The only change git saw was the deletion of the old
0411001 dv1_10 file (replaced by dv1_11 locally).

This wasted significant time and caused Bill justified
frustration. The Phase 2 commit script (Phase2_AA04_
Commit.bat) failed silently because all files were
already tracked.

------------------------------------------------------------
2.4 Why the Premature WD Files Are Not a Future Problem
------------------------------------------------------------

The remaining premature WD files on GitHub (1011001,
1011002, 1011005, 1011007, 1011010, 1011011, 1011012,
1011013, 1011014, and prior 0500000 versions) do NOT
create the same commit problem because:

  - They are already committed and tracked by git
  - Future commits ADD new files or UPDATE existing ones
  - git add [filename] on an already-tracked file that
    has changed WILL stage the change correctly
  - The Phase 2 problem occurred because the files were
    already committed AT THE SAME VERSION as the local
    files — git correctly saw no difference
  - Future updates to WD coding specs (new versions)
    will be detected by git as changes and will stage
    correctly

The only scenario that creates future problems is if
a new file is accidentally committed before its parent
chain exists on GitHub — a document control violation,
not a git mechanics problem. Standing Rule 6 (no git
add .) prevents this.

------------------------------------------------------------
2.5 upstream Now Set — Future Push Syntax
------------------------------------------------------------

The `git push --set-upstream origin master` command set
the tracking relationship between local master and
origin/master. All future pushes from this local clone
use simply:

  git push

No --set-upstream needed again unless the repo is
re-cloned or a new branch is created.

3. DECISIONS MADE THIS SESSION
============================================================

------------------------------------------------------------
3.1 TB00025 Decision Reversed — git add . Was Wrong Call
------------------------------------------------------------

The TB00025 decision to "leave as-is" the premature WD
files was incorrect. The correct action at the time was
to run `git rm --cached` on each premature file to remove
it from GitHub tracking without deleting the local files.
This would have prevented the Phase 2 commit confusion.

This reversal is documented for the record. The premature
files cannot be practically un-committed at this stage
without significant git history rewriting. The current
state is accepted as-is. The lesson is captured below.

------------------------------------------------------------
3.2 Corrected Lessons — Commit Script Standards
------------------------------------------------------------

CORRECTED RULE 6: Never use `git add .` in commit scripts.
  Use explicit `git add [filepath]` per file. Always.
  This rule existed but was not applied to Gate 1 script.
  Root cause: Claude wrote Gate 1 script with git add .
  and Bill ran it without review. Both share responsibility.

NEW RULE 9: After any accidental commit, immediately run
  `git rm --cached [file]` on each accidentally committed
  file and push the correction before the next session.
  Do not defer. Deferral creates compounding problems.

NEW RULE 10: Before running any commit script, Bill runs
  `git status` first to verify what will be committed.
  If unexpected files appear: stop, do not run the script,
  tell Claude.

NEW RULE 11: Future commit scripts include `git status`
  as the first command before any git add, so Bill can
  see the state before changes are staged.
