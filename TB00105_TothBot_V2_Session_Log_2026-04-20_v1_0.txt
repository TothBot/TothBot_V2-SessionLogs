AI-3 (HIGH) — Extend Kraken retry window (OI-020)
------------------------------------------------------------

Objective: Retry schedule covers typical Kraken transient
API outage (observed 2.5-3 minute windows).

Observed Kraken outage data points:
  - 2026-04-19 02:49:21 → 02:51:54 (~2m 33s)
  - 2026-04-20 06:25:46 → 06:28:18 (~2m 32s)
  Both rejected valid credentials with
  `EAPI:Invalid key`. Both recovered without manual
  intervention.

Current schedule (WM-RECONNECT-017):
  N = 10
  base = 1s, cap = 30s, factor = 2x
  Worst-case wall time: 1+2+4+8+16+30+30+30+30+30 = 181s
  Too short by 15-35 seconds for observed outages.

Proposed schedule:
  N = 20 (doubled)
  base = 1s, cap = 30s, factor = 2x (unchanged)
  Worst-case wall time: 181s + 10×30s = 481s (~8 min)
  Covers observed outages with large safety margin.

Rationale:
  - Simplest change (single constant).
  - Keeps proven exponential curve.
  - Matches CIATS parameter philosophy — starting
    value to be replaced by data.
  - Easy to revert if 20 proves too many.
  - Conservative; no behavior regression risk.

Rejected alternatives:
  β (cap 30s → 60s): larger behavior change, two
    constants to tune, less intuitive.
  γ (degraded mode): prolonged outages would go silent.
    Violates Deming 6.5 "system must surface problems."

Affected files:
  1. tothbot/ws_manager.py dv1_20 → dv1_21:
       MAX_RECONNECT_ATTEMPTS: int = 10
     → Change to 20
     Revision history entry.
  2. 1011002 WS_Manager_Coding_Spec dv1_19 → dv1_20:
       WM-RECONNECT-017 attempts value 10 → 20.
       Revision history entry.
  3. 1010000 Trading_System_Coding_Anchor: SpecVer
     update + RegDate.

Deploy verification:

    transient window.
  - Expect RECONNECT_COMPLETE to fire instead of
