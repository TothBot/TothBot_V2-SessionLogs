SEARCH 8 — orjson/Python Decimal precision 2025
------------------------------------------------------------

Sources: orjson GitHub (ijl/orjson), orjson PyPI,
morethanmonkeys Medium article (Aug 2025), blog.krtk.dev
orjson deep dive, benchmarking article DEV community,
DEV community msgspec benchmark (Jul 2025), Medium orjson
vs json (Jan 2026), Dollar Dhingra benchmark, GitHub
orjson-decimal. References: 9 unique.

FINDINGS:
S8-F1: CRITICAL NEW — orjson does NOT natively serialize
       `decimal.Decimal`. It raises `TypeError`:
       `orjson.dumps({"price": Decimal("9.99")})` → TypeError.
       Requires explicit `default` callback:
       ```python
       def default(obj):
           if isinstance(obj, decimal.Decimal):
               return str(obj)
           raise TypeError
       orjson.dumps(data, default=default)
       ```
       Current BP-JSON-002 says "orjson.dumps().decode()"
       but does NOT specify the Decimal default callback.
       TothBot Logger serializes Decimal prices/quantities.
       Without this callback, all Logger serialization of
       trade events will raise TypeError and crash the Logger.
       This is a DEFECT in the current spec.
       Action: Add BP-JSON-005 to 1011001 (dv1_2).
S8-F2: CONFIRMED — orjson 2-10x faster than stdlib json.
       Confirmed correct choice.
S8-F3: CONFIRMED — orjson returns bytes not str.
       BP-JSON-002 "orjson.dumps().decode()" correctly
       handles this.
S8-F4: NOTE — msgspec is an alternative with similar
       performance and native Decimal support. However,
       orjson is already specified and committed to.
       No change to library choice. Only add the callback.

SEARCH 8 CONCLUSION: CRITICAL defect found in 1011001 —
missing orjson Decimal default callback. TothBot Logger
will crash on first trade event without it. Fixed in
1011001 dv1_2 with BP-JSON-005.
