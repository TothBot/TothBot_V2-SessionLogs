------------------------------------------------------------
3.2 Claude violated the single-copy-box rule by adding
    "run in cmd.exe" AFTER issuing the command
------------------------------------------------------------

Claude delivered a Windows-syntax copy box using `&&`
chaining and `copy /Y`. Claude then added a follow-up
instruction: "Run it in cmd.exe (not PowerShell)". Bill
had already run the command in PowerShell before the
follow-up arrived, and PowerShell rejected `&&` as not
a valid statement separator in that version.

Bill corrected:

  "You can't say to run something and then afterwards
   say, Oh by the way, do something else first. I
   already ran the command in power shell. Below is
   the output."

Claude's error: split a single operational unit (the
shell choice plus the command body) across two
responses. The rule that "one copy box is one
copy/paste operation" means the shell context must be
embedded in or determined before the copy box is
issued, not after.

Corrective: Claude reissued the command wrapped in
`cmd /c '...'` so it executed correctly regardless of
whether PowerShell or cmd was the outer shell. The
wrap-with-cmd-/c pattern is the standard for any
`&&`-chained command targeting a PowerShell user.

Lesson: all shell context required to run a copy box
is part of the copy box itself. No separate
"run this in X" instruction permitted.
